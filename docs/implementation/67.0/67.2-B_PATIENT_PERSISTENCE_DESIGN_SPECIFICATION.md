# 67.2-B Patient Persistence Design Specification

**Baseline:** DENTALOPERIX_BASELINE_66_1  
**Autoridad:** Architect Principal / Technical Reviewer / Governance Guardian  
**Fecha de actualización:** 2026-06-24  
**Idioma de comentarios:** Español

## Restricciones vigentes

- No Dual Write.
- No Lead Replacement.
- No New Lead Source of Truth.
- No Persistence Re-Architecture.
- No RBAC Bypass.
- No Automated Patient Merge.
- No modificación de componentes protegidos.

## Componentes protegidos

- BookingDialog
- processDentalLead
- /api/leads/create
- Calendar
- Gmail
- FloatingDentalAIChat
- Home
- siteServices.ts

## Objetivo

Definir la arquitectura de persistencia del dominio Patients reutilizando el patrón certificado.

## Arquitectura objetivo

```text
PatientPersistencePort
    ↓
PatientPersistenceProvider
    ↓
RelationalPatientPersistenceAdapter
    ↓
Supabase PostgreSQL
```

## Principios

- Separación de dominios: Leads != Patients != Appointments.
- Patients es Source of Truth de identidad.
- Reutilización del patrón Port/Provider/Adapter.
- Persistencia desacoplada de marketing, lead lifecycle y scheduling.

## PatientPersistencePort

Contrato abstracto de persistencia. No contiene SQL, ORM, Supabase ni infraestructura.

Responsabilidades: crear, obtener, actualizar, consultar y administrar identidad persistida.

## PatientPersistenceProvider

Punto único de acceso del dominio hacia persistencia. Orquesta persistencia, aplica reglas de dominio y mantiene desacoplamiento.

No puede modificar Leads, Appointments ni ejecutar lógica comercial.

## RelationalPatientPersistenceAdapter

Implementación relacional del contrato. Responsable de persistencia física, lectura, actualización, búsqueda y mapeo de datos.

No decide identidad, no resuelve duplicados y no ejecuta merges.

## Dependencias prohibidas

```text
Patients Domain -> Supabase
Patients Domain -> SQL
Patients Domain -> Infraestructura
```
