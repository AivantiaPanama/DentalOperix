# 67.2-D Patient Identity Governance Specification

**Baseline:** DENTALOPERIX_BASELINE_66_1  
**Autoridad:** Architect Principal / Technical Reviewer / Governance Guardian  
**Fecha de actualización:** 2026-06-24  
**Idioma de comentarios:** Español

## Restricciones vigentes

- No Dual Write.
- No Lead Replacement.
- No New Lead Source of Truth.
- No Persistence Re-Architecture.
- No RBAC Bypass.
- No Automated Patient Merge.
- No modificación de componentes protegidos.

## Componentes protegidos

- BookingDialog
- processDentalLead
- /api/leads/create
- Calendar
- Gmail
- FloatingDentalAIChat
- Home
- siteServices.ts

## Objetivo

Establecer las reglas de gobernanza que controlan identidad del paciente.

## Principio rector

La identidad del paciente es un activo crítico: persistente, auditable, trazable y gobernado.

## Autoridad de identidad

```text
Patients = Autoridad de Identidad
Leads != Autoridad de Identidad
Appointments != Autoridad de Identidad
```

## Componentes de identidad gobernados

- PatientId
- Nombre
- Apellido
- Nombre preferido
- Fecha de nacimiento
- Email
- Teléfono

## Clasificación de cambios

| Tipo | Descripción                  | Impacto |
| ---- | ---------------------------- | ------- |
| A    | Corrección administrativa    | Bajo    |
| B    | Actualización de información | Medio   |
| C    | Cambio de identidad sensible | Alto    |

## Requisitos de auditoría

Toda modificación deberá registrar quién modificó, cuándo, qué campo cambió, valor anterior, valor nuevo y motivo del cambio.

## Gestión de duplicados

Automated Patient Merge permanece prohibido. Se permiten detección, marcado, clasificación y revisión manual.

## Relación con Leads

No está permitido modificar identidad del paciente utilizando estados comerciales del Lead.

## Relación con Appointments

Appointments puede referenciar PatientId, pero no puede modificar identidad.

