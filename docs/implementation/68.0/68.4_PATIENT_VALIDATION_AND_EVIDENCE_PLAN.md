# DENTALOPERIX 68.4 — Patient Validation & Evidence Plan

**Status:** CLOSED / CERTIFIED  
**Governance Mode:** Architect Principal · Technical Reviewer · Governance Guardian  
**Baseline:** DENTALOPERIX_BASELINE_69_2

## 1. Objetivo

Definir el marco de validación y evidencias para certificar cada incremento de implementación del dominio **Patients**.

## 2. Análisis arquitectónico

La validación se realiza sobre:

```text
PatientPersistencePort
→ PatientPersistenceProvider
→ RelationalPatientPersistenceAdapter
→ Supabase PostgreSQL
```

y confirma la separación:

```text
Leads = Acquisition / Marketing / Lead Lifecycle
Patients = Person Identity
Appointments = Scheduled Operational Events
```

## 3. Dependencias bajo validación

- Patient Domain
- PatientPersistencePort
- PatientPersistenceProvider
- RelationalPatientPersistenceAdapter
- Modelo de persistencia Patients
- Esquema correspondiente en Supabase

Deben permanecer invariantes:

- LeadPersistencePort
- LeadPersistenceProvider
- RelationalLeadPersistenceAdapter
- BookingDialog
- processDentalLead
- /api/leads/create
- Calendar
- Gmail
- FloatingDentalAIChat
- Home
- siteServices.ts

## 4. Niveles de validación

| Nivel | Validación   | Evidencia requerida                               |
| ----- | ------------ | ------------------------------------------------- |
| 1     | Arquitectura | Diagrama y revisión técnica                       |
| 2     | Persistencia | Resultados CRUD y transacciones                   |
| 3     | Dominio      | Casos funcionales y revisión del modelo           |
| 4     | Integración  | Matriz de dependencias y revisión de consumidores |
| 5     | Gobernanza   | Checklist e informe de compatibilidad             |

## 5. Criterios de aceptación

- Todos los controles ejecutados.
- Evidencias disponibles y reproducibles.
- Sin hallazgos críticos abiertos.
- Sin desviaciones de arquitectura.
- Aislamiento de dominios preservado.

## 6. Criterios de rechazo

- Acceso directo al Adapter.
- Modificación de LeadPersistence.
- Violación de Sources of Truth.
- Dependencias circulares.
- Cambios en componentes protegidos.
- Dual Write.
- Persistence Re-Architecture.
- RBAC Bypass.
- Automated Patient Merge.

## 7. Impacto técnico

El plan agrega proceso formal de validación y evidencia. No genera cambios funcionales por sí mismo.

Compatible con certificaciones 57.x–68.3.
