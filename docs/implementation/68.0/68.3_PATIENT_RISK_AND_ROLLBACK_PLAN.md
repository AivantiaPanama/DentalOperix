# DENTALOPERIX 68.3 — Patient Risk & Rollback Plan

**Status:** CLOSED / CERTIFIED  
**Governance Mode:** Architect Principal · Technical Reviewer · Governance Guardian  
**Baseline:** DENTALOPERIX_BASELINE_69_2

## 1. Objetivo

Definir el marco de gestión de riesgos y rollback para la implementación del dominio **Patients**, garantizando reversibilidad sin comprometer Leads, Appointments ni componentes protegidos.

## 2. Análisis arquitectónico

El rollback se limita al dominio Patients:

```text
PatientPersistencePort
→ PatientPersistenceProvider
→ RelationalPatientPersistenceAdapter
→ Supabase PostgreSQL
```

No se permite rollback conjunto entre dominios ni afectación a LeadPersistence.

## 3. Dependencias afectadas

Directamente involucradas:

- PatientPersistencePort
- PatientPersistenceProvider
- RelationalPatientPersistenceAdapter
- Esquema de persistencia Patients en Supabase

No afectadas:

- LeadPersistencePort
- LeadPersistenceProvider
- RelationalLeadPersistenceAdapter
- BookingDialog
- processDentalLead
- /api/leads/create
- Calendar
- Gmail
- FloatingDentalAIChat
- Home
- siteServices.ts

## 4. Matriz de riesgos

| ID  | Riesgo                               | Probabilidad | Impacto | Mitigación                                      |
| --- | ------------------------------------ | ------------ | ------- | ----------------------------------------------- |
| R1  | Acoplamiento accidental Lead–Patient | Baja         | Alto    | Separación estricta por Ports                   |
| R2  | Escritura fuera del dominio Patients | Baja         | Alto    | Validación de Provider y Adapter                |
| R3  | Corrupción de datos Patients         | Baja         | Alto    | Validaciones transaccionales y evidencia previa |
| R4  | Modificación involuntaria de Leads   | Muy baja     | Crítico | Aislamiento completo de persistencia            |
| R5  | Bypass del Provider                  | Baja         | Alto    | Revisiones de arquitectura y código             |
| R6  | Cambios en componentes protegidos    | Muy baja     | Crítico | Prohibición explícita y revisión de gobernanza  |
| R7  | Incumplimiento de Sources of Truth   | Muy baja     | Crítico | Validación documental y funcional               |

## 5. Condiciones de rollback obligatorio

- Violación de Source of Truth.
- Escritura fuera del dominio Patients.
- Acceso directo al Adapter sin Port/Provider.
- Dependencia circular.
- Cambios en componentes protegidos.
- Dual Write.
- Persistencia inconsistente.
- Pérdida de integridad referencial.
- Incumplimiento de la arquitectura certificada.

## 6. Condiciones sin rollback

No requiere rollback un error previo a despliegue, una prueba fallida sin impacto persistente o un defecto corregible sin modificar estado persistido.

## 7. Impacto técnico

La ejecución futura deberá incorporar controles de rollback específicos para Patients. No existe impacto autorizado sobre Leads, Appointments ni componentes protegidos.

## 8. Compatibilidad 57.x–68.2

Compatible con certificaciones 57.x–68.2.
