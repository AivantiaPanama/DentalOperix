# DENTALOPERIX 68.2 — Patient Implementation Strategy

**Status:** CLOSED / CERTIFIED  
**Governance Mode:** Architect Principal · Technical Reviewer · Governance Guardian  
**Baseline:** DENTALOPERIX_BASELINE_69_2

## 1. Objetivo

Definir la estrategia de implementación de la persistencia **Patients** dentro de la arquitectura certificada, sin iniciar código durante la fase de planificación original.

Esta fase respondió a: **¿Cómo se ejecutará la implementación?**

## 2. Alcance

Cubre únicamente el dominio **Patients**. No modifica Leads, Booking, Calendar, Gmail, AI, Marketing ni Appointments.

## 3. Análisis arquitectónico

Arquitectura Patients certificada:

```text
PatientPersistencePort
→ PatientPersistenceProvider
→ RelationalPatientPersistenceAdapter
→ Supabase PostgreSQL
```

Arquitectura Leads preservada:

```text
LeadPersistencePort
→ LeadPersistenceProvider
→ RelationalLeadPersistenceAdapter
→ Supabase PostgreSQL
```

## 4. Dependencias afectadas

Nuevas dependencias autorizadas para implementación futura:

- Patient Domain
- PatientPersistencePort
- PatientPersistenceProvider
- RelationalPatientPersistenceAdapter
- Patient Repository
- Patient Services
- Supabase PostgreSQL, únicamente dentro del dominio Patients

Dependencias invariantes:

- LeadPersistencePort
- LeadPersistenceProvider
- RelationalLeadPersistenceAdapter
- BookingDialog
- processDentalLead
- /api/leads/create
- Calendar
- Gmail
- FloatingDentalAIChat
- Home
- siteServices.ts

## 5. Estrategia por etapas

1. Infraestructura de persistencia: Port, Provider, Adapter e interfaces.
2. Repository de Patients: persistencia únicamente, sin lógica de negocio externa.
3. Servicios de dominio: Create, Read, Update Patient.
4. Integración con application layer sin tocar componentes protegidos.
5. Consumo controlado exclusivamente vía `PatientPersistencePort`.

## 6. Reglas de implementación

- Nunca acceder directamente al Adapter.
- Siempre usar Port → Provider → Adapter.
- No reutilizar objetos Lead como Patient.
- No compartir tablas.
- No crear sincronización automática Lead→Patient.
- No introducir Dual Write.

## 7. Riesgos

| Riesgo                               | Mitigación                                   |
| ------------------------------------ | -------------------------------------------- |
| Acoplamiento accidental Lead–Patient | Separación estricta por Ports                |
| Violación de Source of Truth         | Patients mantiene únicamente Person Identity |
| Dependencias circulares              | Ports unidireccionales                       |
| Bypass del Provider                  | Revisión arquitectónica obligatoria          |

## 8. Impacto técnico

Impacto limitado al dominio Patients. No hay impacto autorizado sobre Leads, Appointments, Calendar, Gmail, Booking ni componentes protegidos.

## 9. Compatibilidad 57.x–68.1

Compatible con la arquitectura certificada 57.x–68.1. No requiere re-arquitectura ni excepción de gobernanza.
