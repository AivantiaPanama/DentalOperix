# DentalOperix — WP-02 I1-M5 Clinical Note API & Contracts Report

## Estado

**IMPLEMENTED / PENDING USER VALIDATION**

## Propósito

Exponer la capacidad **Clinical Notes Foundation** mediante una capa API controlada, conectada a la Application Layer certificada y sin introducir dependencias de UI ni modificar dominios protegidos.

## Alcance implementado

### API Contracts

- `registerClinicalNoteApiSchema`
- `updateClinicalNoteApiSchema`
- `clinicalNoteJsonResponse`
- `readClinicalNoteJsonPayload`
- parsing controlado de errores API
- extracción controlada de parámetros `patientId` y `clinicalNoteId` desde la ruta

### API Controller

- `ClinicalNoteApiController`
- registro de notas clínicas
- consulta de historial por paciente
- consulta de detalle de nota clínica
- operaciones de transición sobre nota clínica:
  - complete
  - reopen
  - amend
  - archive

### Application Layer Extension

Se incorporaron casos de uso de consulta necesarios para soportar los endpoints de lectura:

- `GetClinicalNoteUseCase`
- `ListClinicalNotesByPatientUseCase`

Además, las operaciones de modificación aceptan `patientId` opcional para validar que la nota clínica pertenece al paciente de la ruta antes de permitir la transición.

### Routes

Se agregaron rutas API:

- `GET /api/clinical-records/{patientId}/notes`
- `POST /api/clinical-records/{patientId}/notes`
- `GET /api/clinical-records/{patientId}/notes/{clinicalNoteId}`
- `PUT /api/clinical-records/{patientId}/notes/{clinicalNoteId}`

## Seguridad y permisos

- Lectura: `clinical:read`
- Escritura: `clinical:write`

La API no expone endpoints sin validación RBAC.

## Restricciones respetadas

- No se modificó UI.
- No se modificó Leads.
- No se modificó Patients.
- No se modificó Appointments.
- No se alteró la baseline.
- No se introdujo dual write.
- No se reabrieron decisiones arquitectónicas certificadas.
- No se modificaron componentes protegidos.

## Resultado arquitectónico

La capa API consume exclusivamente la Application Layer y no contiene reglas clínicas de dominio.

```text
Route
  -> API Controller
    -> Application Service
      -> Use Case
        -> Repository Port
          -> Infrastructure Adapter
            -> Domain Entity
```

## Validación pendiente

La validación técnica debe ser ejecutada por el usuario conforme a la política vigente del proyecto:

- `npm run build`

I1-M5 deja la capacidad Clinical Notes expuesta mediante contratos API controlados, preservando la separación entre rutas HTTP, aplicación, dominio e infraestructura.
