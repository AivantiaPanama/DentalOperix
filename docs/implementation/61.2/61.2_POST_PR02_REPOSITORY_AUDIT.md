## Important audit finding: documentation drift in PR-61.2-02 summaries

The implementation summaries in `docs/implementation/61.2/61.2_CODE_AUDIT_AND_UPDATE_REPORT.md` and `docs/implementation/61.2/PR-61.2-01_02_IMPLEMENTATION_SUMMARY.md` state or imply that Today's Schedule consumes `/api/leads/list` / Leads as the read source.

Actual code does not do that.

Actual `TodayScheduleWidget.tsx` source:

- imports `useAppointments` from `@/lib/appointments-store`
- imports `Appointment` from `@/lib/clinic-data`
- filters local app appointments by `date`, `time`, and `status !== "cancelled"`

Therefore, the code-accurate statement is:

```text
PR-61.2-02 Today's Schedule currently consumes the existing frontend appointment store (`useAppointments`) and not `/api/leads/list`.
```

Governance interpretation:

- This is acceptable for PR-61.2-02 if treated as a read-only UI schedule view.
- It must not be described as consuming certified Leads persistence unless the code is changed in a future authorized PR.
- PR-61.2-03 Lead Queue must consume certified lead read surfaces, not the appointment store.
