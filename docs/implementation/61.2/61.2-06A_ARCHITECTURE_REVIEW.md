# 61.2-06A Architecture Review - Appointment Domain Discovery Gate

Status: PROPOSED / GATE REQUIRED
Date: 2026-06-23
Program: 61.2 Assistant / Front Desk Workspace

## 1. Architecture Review

61.2-06A formalizes Appointment Domain Discovery as a mandatory gate before Appointment Operations. The current system has booking, calendar, and local appointment read surfaces, but does not yet contain a complete certified appointment domain suitable for multi-provider clinical operations.

The review confirms that current booking behavior is lead-centered and certified under the existing Leads architecture. Appointment operations must not replace, bypass, or dilute Leads as Source of Truth for acquisition/booking origin.

The appointment domain must be treated as an adjacent operational domain, not as a replacement for Leads.

## 3. Risks

### R1 - Incorrect global time-slot blocking

Current booking UI blocks a date/time globally when any appointment exists at that time. This fails for clinics with multiple available doctors/providers.

### R2 - Appointment Request conflated with Confirmed Appointment

Current terminology risks treating a patient request as a confirmed appointment. This is not adequate when requested time is unavailable but alternatives may exist.

### R3 - Local appointment store as operational source

Current Assistant Today's Schedule consumes a localStorage-based appointment store. This cannot be considered a certified operational appointment source of truth.

### R4 - Provider display heuristic mistaken for provider assignment

TodayScheduleWidget resolves provider from service name. This is display-only and not evidence of a real appointment-provider relationship.

### R5 - Audit gap

The system does not currently record who created, modified, or cancelled an appointment operation.

### R6 - Hidden scope expansion

Provider availability could accidentally drift into Doctor <-> Patient assignment or Patient Management. Those remain deferred.

## 4. Technical Impact

The gate itself has no runtime impact. Future implementation is expected to have moderate to high technical impact because it may require a durable appointment model, migration, API contract, RBAC enforcement, and revised availability logic.

The impact should remain isolated from certified Leads persistence.

## 5. Architectural Options

### Option A - Continue directly with Appointment Operations

Rejected. The current appointment assumptions are not sufficiently validated for multi-provider operation.

### Option B - Patch BookingDialog slot logic only

Rejected for now. It may fix a visible symptom but not the appointment domain gaps: persistence, lifecycle, audit, and provider availability.

### Option C - Introduce 61.2-06A Appointment Domain Discovery Gate

Accepted. This allows DentalOperix to document current behavior, identify gaps, and define correct appointment domain foundations before implementation.

## 6. Technical Recommendation

Proceed with 61.2-06A as mandatory gate. Do not implement Appointment Operations until the following are approved:

1. Appointment Domain Discovery Report.
2. Appointment ADR package.
3. Appointment Foundation architecture.
4. File-by-file implementation plan for 61.2-06B.
5. Test strategy updated for multi-provider availability and appointment request lifecycle.

## 8. Test Strategy

No runtime tests are required for documentation-only 61.2-06A.

Future 61.2-06B/06C must include tests for:

- Multiple providers available at the same date/time.
- Same provider conflict at overlapping date/time.
- Alternative recommendation when exact requested slot is unavailable.
- Appointment request saved when patient does not confirm alternatives.
- Assistant review queue visibility.
- Audit trail for create/update/cancel.
- Physical delete absence.
- Regression on Lead Queue, Lead Detail, Status Management, Notes, Today's Schedule.

## 9. Documentation Strategy

61.2-06A documents a domain gate, not a feature delivery. The ADR package captures accepted direction. The architecture assessment register captures evidence for future evaluation of incremental evolution vs DentalOperix Next.
