## 1. Architecture Review

PR-61.2-06 debe permitir que el Front Desk gestione citas desde el Assistant Workspace sin reabrir la arquitectura certificada de Leads.

El alcance funcional recomendado es:

1. Crear cita desde contexto de Lead.
2. Modificar cita existente.
3. Cancelar cita existente.
4. Registrar auditoría de quién creó, modificó o canceló la cita.
5. Validar disponibilidad por recurso clínico asignado, inicialmente doctor/proveedor.

### Decisión conceptual

Crear una cita desde Assistant Workspace no debe confundirse con abrir expediente clínico.

Definiciones:

```text
Appointment = entidad operativa de agenda
Booking = solicitud o agendamiento iniciado por paciente/canal externo
Appointment Operation = acción interna ejecutada por usuario autorizado de clínica
```

Por lo tanto:

```text
Crear cita = reservar un espacio de atención en agenda
Crear cita != abrir expediente clínico
Crear cita != crear relación clínica permanente Doctor ↔ Patient
```

### Disponibilidad correcta

El sistema no debe validar disponibilidad únicamente por fecha y hora.

Modelo incorrecto:

```text
date + time = unique slot
```

Modelo correcto mínimo:

```text
date + time + providerId = unique provider slot
```

Esto permite escenarios reales:

```text
10:00 Dr. A ocupado
10:00 Dr. B disponible
10:00 Dr. C disponible
```

En ese escenario, el sistema debe permitir otra cita a las 10:00 si existe un doctor/proveedor disponible.

### Auditoría obligatoria

Cada operación de cita debe registrar actor, rol, canal y momento.

Campos conceptuales mínimos:

```ts
createdByUserId;
createdByRole;
createdVia;
createdAt;
updatedByUserId;
updatedByRole;
updatedVia;
updatedAt;
cancelledByUserId;
cancelledByRole;
cancelledVia;
cancelledAt;
cancellationReason;
```

La implementación exacta dependerá del modelo actual de appointments/calendar y debe confirmarse mediante inspección técnica antes de código.

## 3. Riesgos

### Riesgo 1 — Confundir cita con expediente

Si “crear cita” se interpreta como abrir expediente, se invade Patient Management.

**Mitigación:** documentar que PR-61.2-06 solo gestiona agenda operativa.

### Riesgo 2 — Bloqueo incorrecto por fecha/hora

Si el sistema sigue usando `date + time` como slot único, bloqueará capacidad real de múltiples doctores.

**Mitigación:** validar conflicto por `date + time + providerId`.

### Riesgo 3 — Introducir Doctor ↔ Patient Assignment permanente

Asignar doctor a una cita no debe crear relación clínica permanente entre doctor y paciente.

**Mitigación:** limitar el alcance a `Appointment → Provider`, no `Patient → Doctor`.

### Riesgo 4 — Trazabilidad insuficiente

Si no se registra quién hizo la operación, no habrá auditoría clínica/operativa.

**Mitigación:** campos de audit trail obligatorios.

### Riesgo 5 — Modificar componentes protegidos

Calendar y BookingDialog están protegidos.

**Mitigación:** inspección previa y diseño por integración controlada.

## 5. Opciones arquitectónicas

### Opción A — Reusar flujo paciente/BookingDialog

El asistente seguiría el mismo proceso que un paciente.

**Ventajas:** máxima reutilización visual.

**Desventajas:**

- Mezcla operación interna con booking externo.
- Puede ocultar auditoría del usuario interno.
- Puede depender de componente protegido.
- No diferencia `createdVia = public_booking` de `createdVia = assistant_workspace`.

**Evaluación:** no recomendada como primera opción.

### Opción B — Crear Appointment Operations en Assistant Workspace

El asistente crea/modifica/cancela desde Lead Detail usando servicios de citas existentes o wrappers nuevos.

**Ventajas:**

- Diferencia operación interna de booking externo.
- Permite auditoría explícita.
- Permite validar provider availability.
- Evita modificar BookingDialog.

**Desventajas:**

- Requiere diseño UI/API específico.
- Puede requerir migración de appointments.

**Evaluación:** recomendada.

### Opción C — Redirigir a Calendar existente

El asistente sale del Workspace y usa Calendar.

**Ventajas:** menor alcance inicial.

**Desventajas:**

- Menor eficiencia operativa.
- Riesgo de tocar Calendar protegido.
- Trazabilidad puede quedar incompleta.

**Evaluación:** viable solo como fallback.

---

## 6. Recomendación técnica

Adoptar Opción B.

Definir PR-61.2-06 como:

> Appointment Operations desde Assistant Workspace: crear, modificar y cancelar citas desde contexto de Lead, registrando audit trail de usuario interno y validando disponibilidad por doctor/proveedor.

Reglas obligatorias:

1. Una cita es agenda operativa, no expediente.
2. El paciente puede agendar y el asistente puede crear cita, pero los canales deben diferenciarse.
3. Toda cita debe registrar quién la creó/modificó/canceló.
4. La disponibilidad se valida por provider, no solo por fecha/hora.
5. `Appointment → Provider` está permitido como asignación operativa de cita.
6. `Patient → Doctor` permanente queda diferido.
7. `Lead ↔ Patient` queda diferido.
8. No modificar componentes protegidos sin autorización explícita.

## 8. Estrategia de pruebas

### Pruebas unitarias

1. Crear cita con provider disponible.
2. Rechazar cita cuando el mismo provider ya está ocupado.
3. Permitir cita en misma fecha/hora con provider diferente.
