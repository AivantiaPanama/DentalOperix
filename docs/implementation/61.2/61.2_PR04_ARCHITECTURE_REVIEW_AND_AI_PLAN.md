## Architecture Review

### Review result

```text
PASS for PR-61.2-03 continuation.
CONDITIONAL HOLD for PR-61.2-04 if scope includes status/notes mutation.
```

PR-61.2-04 is identified by the functional package as `US-61.2-04 — Update a lead's status`, paired with `US-61.2-05 — Update a lead's notes` and governed by `BR-61.2-002 — Lead Update Attribution and Failure Integrity`.

Those stories require persisted, attributable lead updates. The current certified persistence port supports:

```text
LeadPersistenceUpdateInput:
- status
- calendarEventId
- emailSent
```

It does not currently expose a certified `notes` update field. Therefore, implementing status-only may be possible inside the certified Leads persistence path, but implementing notes requires explicit architecture approval for how notes map to the certified Leads schema, likely the existing `message` column/field or a formally added field.

### Governance concern discovered

The repository also contains operational lead surfaces:

```text
src/components/assistant/LeadOperationsWorkspace.tsx
src/routes/api/leads/operations.ts
src/routes/api/leads/$id/operations.ts
src/server/leads/operations-repository.ts
```

These use an auxiliary `.data/lead-operations.json` store for operational status, priority, contact result, internal note, and attribution. This must not be promoted into PR-61.2-04 without Architecture Review because it may be interpreted as a separate operational source for lead updates.

### Architecture decision required before code

Before implementing PR-61.2-04, choose one of the following:

1. **PR-61.2-04A — Safe read-only Lead Detail**
   - No mutation.
   - Shows selected lead details from `/api/leads/list` only.
   - Preserves PR-61.2-03 architecture.
   - Can proceed without persistence change.

2. **PR-61.2-04B — Status update only**
   - Uses existing `LeadPersistencePort.updateLead(id, { status })`.
   - Requires a new governed API route with `leads:update` permission.
   - Does not implement notes.
   - Needs test coverage for RBAC, failure integrity, and protected field rejection.

3. **PR-61.2-04C — Status + notes update**
   - Requires formal architecture decision for notes persistence.
   - Requires updating the certified persistence contract and relational adapter mapping.
   - Must preserve Leads as Source of Truth.
   - Must not use `.data/lead-operations.json` as the source of truth for lead status/notes.

## Riesgos

1. **New Source of Truth risk**
   - Using `.data/lead-operations.json` for lead status or notes would risk violating `Leads = Source of Truth`.

2. **Persistence contract drift**
   - Notes are required by functional docs but not currently present in `LeadPersistenceUpdateInput`.

3. **RBAC vocabulary drift**
   - Docs reference `lead.status.update` and `lead.notes.update`; code currently uses `leads:update`.

4. **Partial update risk**
   - `BR-61.2-002` requires status/notes to be persisted atomically when saved together.

5. **Protected component risk**
   - `/api/leads/create`, `BookingDialog`, `processDentalLead`, `Calendar`, `Gmail`, `FloatingDentalAIChat`, `Home`, and `siteServices.ts` must remain untouched.

6. **Validation environment risk**
   - Local tests cannot run until dependencies are installed. Current ZIP does not include `node_modules`; `vitest` is unavailable until `npm ci` is executed.
