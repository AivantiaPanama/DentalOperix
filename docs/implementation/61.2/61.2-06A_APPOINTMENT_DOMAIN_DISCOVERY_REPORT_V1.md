## 1. Executive Summary

61.2-06A was created as a blocking discovery gate before designing or implementing Appointment Operations. The evidence review confirms that the current codebase contains booking, calendar, and schedule surfaces, but it does not yet contain a complete certified Appointment Domain suitable for multi-provider clinic operations.

The current implementation is lead-centered: the patient booking flow creates a Lead, attempts to create a Google Calendar event, updates the Lead with `calendarEventId` and status `agendada` when Calendar succeeds, and writes a browser-local appointment entry for UI purposes. Today's Schedule in the Assistant Workspace reads from that local appointment store.

The main architectural finding is confirmed: current availability is effectively modeled as `date + time = blocked slot` in the patient booking UI. It does not account for provider/doctor/resource capacity. This would block valid appointments in clinics where multiple doctors can attend patients at the same date/time.

## 4. Detailed Findings

### F-01 — The current appointment model is local and minimal

`src/lib/clinic-data.ts` defines:

```ts
export type Appointment = {
  id: string;
  name: string;
  email: string;
  phone: string;
  service: string;
  date: string;
  time: string;
  status: "pending" | "confirmed" | "cancelled";
  notes?: string;
};
```

This model does not include provider/resource assignment, duration, lead reference, calendar event reference, request lifecycle, audit metadata, source channel, cancellation reason, or assistant review fields.

### F-02 — Local appointment persistence uses browser localStorage

`src/lib/appointments-store.ts` uses key:

```ts
dentaloperix_appointments_v1
```

and exposes local functions `add`, `cancel`, `remove`, and `getBookedSlots(date)`. This is useful for UI/demo behavior but cannot be considered a certified appointment source of truth for production operations.

### F-03 — Availability currently blocks by date/time only

`getBookedSlots(date)` filters non-cancelled appointments by date and returns only `time`:

```ts
.filter((a) => a.date === date && a.status !== "cancelled")
.map((a) => a.time)
```

`BookingDialog.tsx` then disables the slot when:

```ts
const isBooked = booked.includes(t);
const disabled = isBooked || isPast;
```

This confirms the problematic assumption:

```text
date + time = blocked slot
```

Correct clinic behavior requires at least:

```text
date + time + provider/resource = availability
```

### F-04 — Provider display is heuristic, not assignment

`TodayScheduleWidget.tsx` contains a static `providerByService` map:

```ts
"Diseño de Sonrisa" -> "Dra. Camila Ríos"
Ortodoncia -> "Dr. Felipe Soto"
...
```

This supports display, but it is not a persisted provider assignment. There is no `providerId`, `doctorId`, provider calendar, provider capacity, or provider conflict check attached to the appointment entity.

### F-05 — Patient booking creates a Lead first

`src/lib/api/dental.functions.ts` validates appointment-like form data, generates an `appointmentId`, and calls protected `processDentalLead()`.

`processDentalLead()` builds a Lead record with:

- `id = appointmentId`
- `preferredDate = date + time`
- `status = nuevo`
- `calendarEventId = ""`
- `emailSent = false`

Then it appends the Lead via the certified Lead persistence adapter. This confirms that patient booking remains lead-centered and compatible with Leads = Source of Truth.

### F-06 — Google Calendar event creation has no availability or provider conflict check

`src/lib/google/google.server.ts` and `src/server/google/calendar.ts` create Calendar events using date/time and fixed duration. No provider-level calendar selection, resource lookup, conflict query, availability recommendation, or provider attendee requirement was found.

There are two Calendar creation paths:

1. `src/lib/google/google.server.ts#createGoogleCalendarEvent()` used by `processDentalLead()`.
2. `src/server/google/calendar.ts#createDentalAppointment()` exposed through `/api/calendar/create-event`.

Both create Google Calendar events, but neither implements a certified Appointment Domain lifecycle.

### F-07 — Assistant Today's Schedule is read-only and local-store based

`TodayScheduleWidget.tsx` consumes `useAppointments()` and filters today's non-cancelled appointments. It does not fetch production appointments, Calendar events, or provider availability. It is a read-only operational surface prepared for Front Desk, not an operational appointment system.

### F-08 — Appointment Request lifecycle is absent

No model was found for:

- `requested`
- `suggested_alternative`
- `pending_patient_confirmation`
- `needs_assistant_review`
- `expired`

Current local appointment states are only `pending`, `confirmed`, and `cancelled`. Current Lead CRM states are not sufficient to model appointment requests without conflating Lead lifecycle and Appointment lifecycle.

### F-09 — Audit trail is absent

No appointment operation audit fields were found:

- `createdByUserId`
- `createdByRole`
- `createdVia`
- `updatedByUserId`
- `cancelledByUserId`
- `cancelledVia`
- `cancellationReason`

This is a required gap before assistant-side appointment operations.

### F-10 — Execution Continuity Rule is now present in governance backlog

`docs/governance/61.2_GOVERNANCE_RETROSPECTIVE_CANDIDATES.md` includes `GRC-005 - Execution Continuity Rule`, with origin in 61.2-06A. This confirms the governance improvement is captured for 61.2 retrospective review.

## 6. Architectural Conclusion

The appointment domain is currently distributed across three different surfaces:

```text
Lead persistence
  - certified acquisition/booking origin record
  - stores preferredDate, CRM status, calendarEventId

Google Calendar
  - external calendar event
  - created after Lead persistence succeeds

Local appointment store
  - browser-local UI appointment list
  - feeds BookingDialog blocked slots and Assistant Today's Schedule
```

This is not yet a safe foundation for Assistant Appointment Operations.

The current code confirms that PR-61.2-06 must not proceed directly as create/modify/cancel UI work. A separate Appointment Foundation is required first.

## 7. Recommendation for 61.2-06B

Open `61.2-06B Appointment Foundation` before Assistant Appointment Workflow.

Minimum architecture scope:

1. Define durable Appointment / Appointment Request model.
2. Define provider/resource availability model.
3. Define appointment lifecycle states.
4. Define audit fields for create/update/cancel.
5. Define relationship boundaries with Leads and Calendar.
6. Define whether local appointment store is deprecated, adapted, or isolated as demo-only.
7. Define Calendar integration boundary without modifying protected Calendar behavior prematurely.
8. Define alternative recommendation policy when exact date/time is unavailable.
9. Define assistant review queue for unconfirmed requests.

Do not implement code until the 61.2-06B Architecture Review, file-by-file implementation plan, and test strategy are approved.

## 8. Risks

### R1 — Invalid capacity blocking

If the system continues using global date/time blocking, it will reject valid appointments whenever another provider is available at the same time.

### R2 — Local storage mistaken for source of truth

Assistant schedule may appear operational but is local-only. This can mislead future development if not explicitly bounded.

### R3 — Lead lifecycle and appointment lifecycle conflation

Lead status `agendada` is not equivalent to a full appointment lifecycle. Appointment requests, alternatives, confirmations, cancellations, and expirations require their own lifecycle.

### R4 — Protected component pressure

The obvious symptom is in `BookingDialog`, but modifying it directly would violate current governance unless a scoped Architecture Review approves the change.

### R5 — Hidden Doctor ↔ Patient modeling

Provider assignment to a specific appointment is acceptable as scheduling metadata. It must not become a permanent Doctor ↔ Patient assignment without a separate Architecture Review.

## 9. Out of Scope

61.2-06A does not authorize:

- Code changes.
- BookingDialog modification.
- processDentalLead modification.
- Calendar modification.
- New appointment table or migration.
- Patient Management.
- Doctor ↔ Patient permanent assignment.
- Lead ↔ Patient relationship model.
- Retention or soft delete policy.
