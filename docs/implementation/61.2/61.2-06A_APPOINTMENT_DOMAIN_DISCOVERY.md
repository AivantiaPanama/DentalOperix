# 61.2-06A Appointment Domain Discovery Report

Status: PROPOSED / REQUIRED GATE
Program: 61.2 Assistant / Front Desk Workspace
Date: 2026-06-23
Owner: Architecture Governance

## Executive Summary

61.2-06A is a mandatory discovery gate before designing or implementing Appointment Operations. The review found that the current implementation contains appointment-related behavior, but it is not yet a complete operational appointment domain for a multi-provider clinic.

The current implementation supports a patient-facing booking flow that creates a Lead, attempts Google Calendar event creation, and stores a local front-end appointment entry. It also supports a read-only Assistant Today's Schedule widget based on the local appointment store.

The current implementation does not yet provide a certified appointment persistence layer, provider-level capacity model, appointment audit trail, appointment request lifecycle, or assistant-side appointment operation workflow.

## Current Implementation Findings

### 1. Appointment Data Shape

`src/lib/clinic-data.ts` defines a front-end `Appointment` type with:

```ts
id
name
email
phone
service
date
time
status: "pending" | "confirmed" | "cancelled"
notes?
```

There is no providerId, providerUserId, location/resource identifier, duration field, audit field, request state, or durable appointment persistence contract in this type.

### 2. Local Appointment Store

`src/lib/appointments-store.ts` stores appointments in browser localStorage using key `dentaloperix_appointments_v1`.

It exposes:

```ts
useAppointments()
add(a)
cancel(id)
remove(id)
getBookedSlots(date)
```

Critical finding: `getBookedSlots(date)` returns occupied times only by date and time:

```ts
.filter((a) => a.date === date && a.status !== "cancelled")
.map((a) => a.time)
```

This confirms the current front-end booking availability assumption:

```text
date + time = blocked slot
```

This is insufficient for a multi-provider clinic.

### 3. BookingDialog Availability Behavior

`src/components/site/BookingDialog.tsx` reads blocked times via `getBookedSlots(data.date)` and disables a slot when:

```ts
const isBooked = booked.includes(t);
const disabled = isBooked || isPast;
```

This means if any appointment exists at 10:00 for a selected date, 10:00 is disabled for all patients, regardless of whether another doctor/provider is available.

### 4. Patient Booking Flow

The patient-facing booking flow calls `useCreateDentalAppointment()`, which calls `createDentalAppointment()` in `src/lib/api/dental.functions.ts`.

That server function calls protected `processDentalLead()` and creates a Lead with `appointmentId`, `preferredDate`, and initial status `nuevo`.

`processDentalLead()` then attempts to create a Google Calendar event and updates the Lead to `agendada` if calendar creation succeeds.

Certified current behavior from `DENTALOPERIX_AI_CONTEXT.md`:

```text
Patient books appointment
  -> Lead stored in Supabase
  -> Clinic Calendar event created
  -> Patient receives one confirmation email
  -> Email includes appointment details and invite.ics
  -> UI shows appointment confirmation
```

### 5. Calendar Event Creation

`src/server/google/calendar.ts` creates a Google Calendar event with:

- start: `${date}T${time}:00`
- duration: fixed 45 minutes
- calendarId from config or primary calendar
- summary including service and patient name
- attendees: patient email and clinic notification email

There is no provider-level calendar selection, provider attendee, provider capacity lookup, or conflict check inside this function.

### 6. Assistant Today's Schedule

`src/components/assistant/TodayScheduleWidget.tsx` consumes `useAppointments()` and displays today's non-cancelled appointments.

Provider is resolved by a static service-to-provider map:

```ts
providerByService[appointment.service] ?? "Proveedor por confirmar"
```

This is a display heuristic, not an operational provider assignment.

### 7. RBAC and Planned Appointment Permissions

RBAC documentation includes appointment permissions for create/read/update/cancel. Assistant is allowed to create appointments for any patient and read all appointments.

However, the current code review did not identify a certified Assistant Appointment Operations workflow or durable appointment operation API aligned with these permissions.

### 8. Documentation / Implementation Mismatch

The 61.2 user stories and test package already refer to appointment create/update/cancel and conflict handling by provider. However, the current implementation still uses localStorage read surface and date/time blocking in BookingDialog.

Therefore, 61.2-06 cannot safely proceed directly to implementation without first reconciling the appointment domain.

## Domain Conclusions

### Confirmed Current Assumptions

- Patient booking currently flows through Lead creation.
- Calendar event creation is attempted after Lead creation.
- Assistant Today's Schedule is read-only and local-store based.
- UI availability blocks date/time globally.
- Provider assignment is not persisted as appointment data.
- Appointment audit trail does not exist.
- Appointment request lifecycle is not modeled.

### Corrected Target Assumptions

- Availability must not be evaluated as date + time only.
- Availability must be evaluated against clinical resources, initially provider/doctor.
- A request for a date/time is not always a confirmed appointment.
- Lack of exact availability is not an error; it should trigger alternative recommendations or assistant review.
- Assistant-created appointments and patient-booked appointments may produce the same confirmed appointment entity, but they are different operational actions and must be audited.

## Required Gate Decision

Before Appointment Operations design begins, DentalOperix must define an Appointment Domain Foundation that includes:

1. Durable appointment persistence strategy.
2. Provider/resource availability model.
3. Appointment request lifecycle.
4. Audit trail fields.
5. Assistant operation workflow.
6. Calendar integration boundaries.
7. Conflict and recommendation policy.
