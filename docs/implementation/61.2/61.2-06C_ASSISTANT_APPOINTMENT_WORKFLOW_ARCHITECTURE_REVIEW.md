## 1. Executive Summary

61.2-06C introduces the first controlled Front Desk workflow on top of the validated Appointment Foundation.

The goal is not to rework the public Booking flow and not to modify protected Calendar or Booking components. The goal is to allow an authorized assistant user to work from the existing Assistant Lead Detail experience and create or progress an operational appointment request using the new appointment domain introduced in 61.2-06B.

This PR must remain a workflow layer over the appointment foundation. It must not reopen the Leads architecture, must not replace Leads as Source of Truth, and must not model permanent Doctor <-> Patient or Lead <-> Patient relationships.

## 3. Architecture Review

### 3.1 Current Certified Architecture

The certified Leads persistence architecture remains unchanged:

```text
Leads
  -> LeadPersistencePort
  -> LeadPersistenceProvider
  -> RelationalLeadPersistenceAdapter
  -> Supabase PostgreSQL
```

Leads remains the Source of Truth for lead capture and lead operational status.

Appointments are now a complementary operational scheduling domain. They may reference a Lead through `leadId`, but they do not replace Leads and do not become a new source of truth for lead information.

### 3.2 Current Appointment Foundation

61.2-06B introduced:

- Appointment domain model.
- Appointment statuses:
  - `requested`
  - `suggested_alternative`
  - `pending_patient_confirmation`
  - `confirmed`
  - `needs_assistant_review`
  - `rescheduled`
  - `cancelled`
  - `expired`
- Appointment sources:
  - `public_booking`
  - `assistant_workspace`
  - `system`
  - `manual_import`
- Appointment audit actor model.
- Provider availability validation.
- Repository boundary.
- Relational repository implementation.
- Tests covering domain, service, availability, and repository behavior.

### 3.3 61.2-06C Scope

61.2-06C should expose a controlled assistant workflow that can:

1. Create an appointment request from a selected Lead.
2. Check provider availability for a requested schedule.
3. Confirm an appointment when provider availability is valid.
4. Move the appointment into `needs_assistant_review` when the request cannot be confirmed.
5. Preserve audit trail using `assistant_workspace` as `createdVia` / `updatedVia`.
6. Keep the Lead visible and optionally update Lead status only through certified Lead endpoints if explicitly required.

61.2-06C should not implement full patient self-service scheduling, public booking rewrite, global provider scheduling, calendar write-back, notification resend, or permanent clinical assignment.

## 5. Risks

### R-01: Creating a second lead source of truth

Risk: Appointment records could accidentally duplicate lead data and become an alternate source of truth.

Mitigation:
- Store appointment-specific operational data only.
- Use `leadId` reference where needed.
- Do not synchronize lead identity fields back from Appointment.
- Do not replace Lead status or notes.

### R-02: Confusing Appointment Request with Confirmed Appointment

Risk: Assistant UI may imply that every scheduling attempt becomes a confirmed appointment.

Mitigation:
- UI must distinguish request, review, and confirmed states.
- Confirmation must require provider availability.
- Unavailable exact slot must lead to review or suggestions, not hard failure.

### R-03: Reintroducing date/time-only blocking

Risk: Workflow may check date/time without provider.

Mitigation:
- All confirmation checks must include providerId + startAt + endAt.
- Provider is required for `confirmed` status.

### R-04: Accidental Doctor <-> Patient assignment

Risk: Provider selection for an appointment may be interpreted as permanent clinical assignment.

Mitigation:
- Document and label provider assignment as appointment-level only.
- Do not create patient-provider relationship tables or fields.

### R-05: Touching protected public Booking / Calendar flow

Risk: Developer may try to reuse public Booking or Calendar by modifying protected components.

Mitigation:
- Add new Assistant-specific workflow component and API boundary.
- Do not modify BookingDialog or Calendar.

## 6. Technical Impact

61.2-06C is a moderate-risk workflow PR because it connects a new domain foundation to an existing Assistant UI. It does not require persistence re-architecture and does not reopen 57.x.

Expected changes:

- New API surface for Assistant Appointment Workflow.
- New UI component for appointment request/confirmation card.
- New tests for assistant workflow and appointment API behavior.
- Optional status update to Lead only through existing certified `/api/leads/update-status`, if approved as part of workflow.

No database schema change is expected if 61.2-06B migration already exists and is accepted as the persistence foundation.

## 7. Architectural Options

### Option A - Assistant Workflow over Appointment Domain APIs

Create dedicated `/api/appointments/*` routes for request, availability, confirmation, and cancellation/review transitions.

Pros:
- Clear boundary.
- Reuses appointment domain.
- Does not touch protected components.
- Testable.

Cons:
- Requires new API surface.

### Option B - Direct UI calls to existing Calendar API

Use `/api/calendar/create-event` or existing public booking patterns.

Pros:
- Less initial surface.

Cons:
- Repeats discovered flaw of calendar/event-first scheduling.
- Risks date/time-only assumptions.
- Risks touching protected Calendar/Booking components.
- Does not fully use 61.2-06B foundation.

### Option C - Modify public BookingDialog to support assistant mode

Pros:
- Reuses existing UI.

Cons:
- Violates protected component restriction.
- Mixes patient self-service with assistant internal operations.
- High regression risk.

## 8. Technical Recommendation

Recommended: Option A.

61.2-06C should implement an Assistant-specific workflow using dedicated appointment APIs over the 61.2-06B appointment domain. It should not modify public BookingDialog or Calendar.

Recommended workflow:

```text
Assistant Lead Detail
  -> Appointment Workflow Card
  -> Create Appointment Request
  -> Check Provider Availability
  -> Confirm if available
  -> Needs Assistant Review if not confirmable
```

## 9. Detailed Implementation Plan by File

### 9.1 New API Routes

Recommended additions:

- `src/routes/api/appointments/request.ts`
  - Creates an appointment request from Assistant Workspace.
  - Uses `AppointmentService.createRequest`.
  - Requires source `assistant_workspace`.
  - Captures audit actor.

- `src/routes/api/appointments/check-availability.ts`
  - Validates provider availability using `AppointmentAvailabilityService`.
  - Requires `providerId`, `startAt`, `endAt`.

- `src/routes/api/appointments/confirm.ts`
  - Confirms an appointment if provider availability is valid.
  - Uses `AppointmentService.confirmAppointment`.

Optional only if in scope:

- `src/routes/api/appointments/mark-review.ts`
  - Moves an appointment request to `needs_assistant_review` when not confirmable.

### 9.2 New Server Wiring

- `src/server/appointments/appointment-service-provider.ts`
  - Provides a factory for AppointmentService backed by RelationalAppointmentRepository.
  - Keeps route handlers thin.

If existing provider pattern exists elsewhere, follow that pattern.

### 9.3 New UI Component

- `src/components/assistant/AssistantAppointmentWorkflowCard.tsx`
  - Hosts request / availability / confirmation UI.
  - Receives selected Lead as input.
  - Does not mutate Lead directly.
  - Calls appointment API routes.
  - Shows clear states:
    - request draft
    - checking availability
    - confirmed
    - needs assistant review

### 9.4 Existing UI Integration

## 10. API Contract Draft

### Create Appointment Request

`POST /api/appointments/request`

Request:

```json
{
  "leadId": "lead_123",
  "requestedDate": "2026-06-25",
  "requestedTime": "14:30",
  "durationMinutes": 60,
  "service": "limpieza",
  "patientName": "Lucia",
  "patientEmail": "lucia@example.com",
  "patientPhone": "+507 60000000",
  "notes": "Prefiere horario vespertino"
}
```

Response:

```json
{
  "success": true,
  "appointment": {
    "id": "appt_...",
    "status": "requested"
  }
}
```

### Check Availability

`POST /api/appointments/check-availability`

Request:

```json
{
  "providerId": "doctor_1",
  "startAt": "2026-06-25T14:30:00.000Z",
  "endAt": "2026-06-25T15:30:00.000Z"
}
```

Response:

```json
{
  "success": true,
  "available": true,
  "conflicts": []
}
```

### Confirm Appointment

`POST /api/appointments/confirm`

Request:

```json
{
  "appointmentId": "appt_123",
  "providerId": "doctor_1",
  "scheduledStartAt": "2026-06-25T14:30:00.000Z",
  "scheduledEndAt": "2026-06-25T15:30:00.000Z"
}
```

Response:

```json
{
  "success": true,
  "appointment": {
    "id": "appt_123",
    "status": "confirmed"
  }
}
```
