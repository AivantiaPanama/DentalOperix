# 61.2-06B Appointment Foundation - Detailed Design

Status: PROPOSED / DESIGN ONLY
Date: 2026-06-23
Owner: Architecture Governance
Predecessor: 61.2-06A Appointment Domain Discovery Report v1
Implementation Authorization: NOT GRANTED

## 1. Executive Summary

61.2-06B establishes a durable Appointment Foundation before any Assistant workflow for creating, modifying, or cancelling appointments.

61.2-06A confirmed that the current system has booking/calendar surfaces but no certified Appointment Domain. The current patient booking flow creates a Lead, attempts Google Calendar event creation, stores calendar metadata on the Lead, and writes a local browser appointment entry for UI purposes. Availability is currently blocked globally by `date + time`, which is incorrect for a multi-provider clinic.

61.2-06B must introduce a controlled Appointment Domain that supports:

- Appointment Request vs Confirmed Appointment separation.
- Provider-aware availability.
- Appointment lifecycle statuses.
- Appointment operation audit trail.
- Durable persistence.
- Safe integration with Leads while preserving Leads as Source of Truth for lead intake and commercial follow-up.

This document is a design artifact only. It does not authorize source code changes.

## 4. Architecture Review

### 4.1 Current Architecture Constraints

The certified Lead architecture remains mandatory:

```text
Leads
  -> LeadPersistencePort
  -> LeadPersistenceProvider
  -> RelationalLeadPersistenceAdapter
  -> Supabase PostgreSQL
```

`Leads = Source of Truth` remains intact. Appointment Foundation must not replace Leads, mutate lead intake semantics, or introduce a second source of truth for lead records.

### 4.2 New Domain Boundary

61.2-06B introduces Appointments as a complementary operational domain:

```text
Lead Intake / Commercial Follow-up
  -> Leads Source of Truth

Appointment Scheduling / Operational Capacity
  -> Appointment Domain
```

This is not a Persistence Re-Architecture. It is a new bounded context for appointment scheduling.

### 4.3 Domain Relationship

A Lead may reference an Appointment, and an Appointment may reference a Lead, but neither domain replaces the other.

```text
Lead
  optional calendarEventId today
  future optional appointmentId

Appointment
  optional leadId
  providerId
  lifecycle status
  requested/scheduled time fields
  audit metadata
```

### 4.4 Availability Principle

Availability is not `date + time`.

Availability is:

```text
requested date/time + duration + provider/resource capacity + existing confirmed/held appointments
```

Minimum 61.2-06B foundation:

```text
date + time + providerId
```

## 6. Risks

| Risk | Severity | Mitigation |
|---|---:|---|
| Appointment domain becomes a new source of truth for Leads | High | Appointment may reference Lead but cannot replace Lead intake/persistence. |
| Assistant create/modify/cancel UI starts before foundation | High | 61.2-06B is a blocking foundation before 61.2-06C. |
| Provider assignment becomes Doctor ↔ Patient assignment | High | Appointment provider assignment is per appointment only, not permanent clinical relationship. |
| Calendar becomes authoritative appointment persistence | Medium | Appointment domain should be durable; Calendar remains integration artifact. |
| Double booking during concurrent scheduling | Medium | Add conflict checks and DB constraints/indexes in design. |
| Overdesign delays 61.2 | Medium | 61.2-06B is foundation only; workflow comes later. |

## 8. Options Considered

### Option A - Appointment as independent durable domain

Appointments are persisted in their own table/repository with optional Lead linkage and optional Calendar integration metadata.

Advantages:
- Correct scheduling boundary.
- Supports provider-aware availability.
- Supports lifecycle and audit.
- Enables Assistant workflow without contaminating Lead persistence.

Disadvantages:
- Requires new persistence and migration.
- Requires later integration with existing booking/calendar flows.

### Option B - Appointment as extension of Lead

Add appointment fields directly to Lead persistence.

Advantages:
- Less new infrastructure.

Disadvantages:
- Violates separation of concerns.
- Risks overloading Leads as scheduling source of truth.
- Makes lifecycle and provider capacity difficult.

### Option C - Appointment as Google Calendar only

Use Google Calendar as appointment authority.

Advantages:
- Leverages existing calendar integration.

Disadvantages:
- Calendar does not model request lifecycle, assistant review, audit, provider capacity, or future operational needs cleanly.

## 9. Technical Recommendation

Use Option A.

Appointments should be a new durable bounded context. Leads remain Source of Truth for lead intake and commercial follow-up. Calendar remains an integration/output channel, not the source of truth for appointment lifecycle.

## 10. Appointment Domain Model - Design Fields

This is a design contract, not implementation code.

### 10.1 Appointment

| Field | Required | Purpose |
|---|---:|---|

| provider_id | No/Conditional | Provider assigned to appointment. Required for confirmed scheduled appointment. |

| source | Yes | `public_booking`, `assistant_workspace`, `manual_import`, etc. |
| patient_name | Yes initially | Administrative name snapshot; not Patient entity. |
| patient_email | No | Contact snapshot. |
| patient_phone | No | Contact snapshot. |
| notes | No | Operational appointment notes. |
| calendar_event_id | No | External Google Calendar linkage. |
| created_by_user_id | No | Actor for internal operations. Null for unauthenticated public booking. |
| created_by_role | No | Actor role snapshot. |
| created_via | Yes | Channel/UI/system path. |

| updated_by_role | No | Last updater role snapshot. |

### 10.2 Provider

61.2-06B should not implement full employee management. Provider records can be introduced as minimal operational resources if required.

| Field | Required | Purpose |

| active | Yes | Whether provider can receive appointments. |
| default_slot_minutes | No | Default duration/capacity model. |

Provider is not a Patient assignment and not a full User identity unless future 61.3 links it explicitly.

## 11. Appointment Status Vocabulary

Initial lifecycle:

| Status | Meaning | Consumes Provider Capacity? |
|---|---|---:|
| requested | Request received, not scheduled. | No |
| suggested_alternative | System or assistant suggested alternatives. | No |
| pending_patient_confirmation | Alternative or slot pending patient confirmation. | Optional, if hold policy is later approved |
| confirmed | Appointment confirmed. | Yes |
| needs_assistant_review | Requires front desk follow-up. | No |
| rescheduled | Appointment has been moved or superseded. | No for old slot; Yes for new confirmed slot |
| cancelled | Appointment cancelled. | No |
| expired | Request or hold expired. | No |

61.2-06B should not implement temporary holds unless explicitly approved. If holds are introduced, they require expiration policy.

## 12. Availability Rules

Minimum conflict rule for confirmed appointments:

```text
Provider conflict exists when:
- same provider_id
- existing status consumes capacity
- time intervals overlap
```

A different provider may accept another appointment at the same date/time.

61.2-06B should define the conflict service and persistence contract. 61.2-06C can use it for Assistant workflows.

## 13. Audit Trail Rules

Every appointment operation must preserve actor and channel metadata.

Minimum channels:

- `public_booking`
- `assistant_workspace`
- `system`
- `manual_import`

Minimum actor rules:

- Public unauthenticated booking may have no `created_by_user_id`, but must have `created_via = public_booking`.
- Assistant operations must include authenticated user identity when available.
- Cancellation must record who cancelled and why when initiated internally.

## 15. Testing Strategy

### Unit tests

- Appointment status vocabulary is closed and typed.
- Invalid lifecycle transitions are rejected or explicitly classified.
- Availability conflict checks are provider-aware.
- Confirmed appointments consume provider capacity.
- Requested / needs assistant review do not consume capacity.
- Audit fields are required for internal operations.

### Integration tests

- Appointment can reference Lead without changing Lead persistence contract.
- Same date/time can be available for Provider B when Provider A is booked.
- Same provider cannot be double-booked for overlapping confirmed appointments.

### Regression tests

- Existing Lead persistence tests remain passing.
- PR05 notes contract remains valid.
- `/api/leads/update-status` and `/api/leads/update-notes` behavior remains unchanged.
- BookingDialog remains unchanged in 61.2-06B unless explicitly authorized later.

## 17. Open Questions Before Implementation

These questions should be answered before implementation, unless the team intentionally implements a safe default:

1. Should provider catalog be static seed data, database table, or linked to future Users?
2. Should `pending_patient_confirmation` consume temporary capacity?
3. What default duration should be used when service duration is unknown?
4. Should public booking create only `requested` appointment records first, or continue current Lead-first flow until 61.2-06C/06D?
5. Should Google Calendar event creation occur only after `confirmed` status?

Recommended safe defaults for 61.2-06B:

- Provider table/minimal catalog exists independently of Users.
- Pending confirmation does not consume capacity until hold policy is approved.
- Unknown duration defaults to 60 minutes.
- Public BookingDialog remains unchanged in 61.2-06B.
- Google Calendar integration remains unchanged in 61.2-06B.
