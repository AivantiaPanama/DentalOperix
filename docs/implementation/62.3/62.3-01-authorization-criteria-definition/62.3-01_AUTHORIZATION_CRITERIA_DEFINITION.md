# 62.3-01 Authorization Criteria Definition

## Certified Baseline

- 57.x CLOSED / CERTIFIED.
- 61.1 CLOSED / CERTIFIED.
- 61.2 CLOSED / CERTIFIED.
- 61.3 CLOSED / CERTIFIED.
- 61.4 CLOSED / DISCOVERY CERTIFIED / ARCHITECTURE CERTIFIED.
- 62.0 DOMAIN DESIGN CERTIFIED.
- 62.1 TECHNICAL DESIGN CERTIFIED.
- 62.2 CLOSED / GOVERNANCE CERTIFIED.

## Certified Architecture

```text
Leads -> LeadPersistencePort -> LeadPersistenceProvider -> RelationalLeadPersistenceAdapter -> Supabase PostgreSQL
```

## Certified Sources of Truth

- Leads = Source of Truth for acquisition, marketing, and lead lifecycle.
- Patients = Source of Truth for person identity.
- Appointments = Source of Truth for scheduled operational events.

## Protected Components

BookingDialog, processDentalLead, /api/leads/create, Calendar, Gmail, FloatingDentalAIChat, Home, siteServices.ts.

## Prohibited Actions

Dual Write, Lead Replacement, New Lead Source of Truth, Persistence Re-Architecture, RBAC Bypass, Automated Patient Merge.

## Authorization Boundary

62.3 is documentation, governance, and authorization-review preparation only. It does not authorize implementation, code generation, detailed design, database changes, API changes, UI changes, migration, deployment, or architectural modification.

## Objective

Define the minimum criteria required before a future formal authorization review may be considered.

## Architectural Analysis

The criteria do not modify architecture, introduce components, redefine responsibilities, or alter Sources of Truth.

## Affected Dependencies

None.

## Risks

- Authorization based on incomplete evidence.
- Confusion between readiness and authorization.
- Reopening decisions already certified.

## Technical Impact

None.

## Authorization Criteria Baseline

| Criterion | Description | Current Status |
|---|---|---|
| AC-01 Architecture Compatibility | Future proposals must demonstrate compatibility with certified architecture. | Met |
| AC-02 Source of Truth Protection | No conflict with Leads, Patients, and Appointments ownership boundaries. | Met |
| AC-03 Certification Compatibility | Traceability to 57.x, 61.x, 62.0, 62.1, and 62.2 required. | Met after 62.3-02 |
| AC-04 Protected Component Control | No direct or indirect modification of protected components without future explicit approval. | Met |
| AC-05 Governance Compliance | Restrictions remain enforceable. | Met |
| AC-06 Risk Evidence | Formal risk evaluation required for future authorization review. | Met in 62.3 context |
| AC-07 Technical Impact Evidence | Future request must quantify expected impact. | Template gap identified |
| AC-08 Reversibility Evidence | Rollback or containment required when applicable to a concrete proposal. | Not applicable in documentation phase |

## Exclusion Criteria

A future request cannot advance if it introduces Dual Write, Lead Replacement, New Lead Source of Truth, Persistence Re-Architecture, RBAC Bypass, Automated Patient Merge, protected-component modification without explicit approval, or lacks risk and impact analysis.
