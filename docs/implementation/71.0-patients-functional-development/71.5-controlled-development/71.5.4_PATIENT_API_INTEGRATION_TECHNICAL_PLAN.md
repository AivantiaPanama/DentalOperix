# 71.5.4 — Patient API Integration Technical Plan

## Status

PLANNING AUTHORIZED

## Objective

Expose the Patient domain through controlled HTTP endpoints while reusing certified layers only:

```text
API Routes
  -> Patient Application Layer
  -> PatientPersistencePort
  -> PatientPersistenceProvider
  -> RelationalPatientPersistenceAdapter
  -> Supabase PostgreSQL
```

- POST /api/patients/create
- GET /api/patients/:id
- GET /api/patients/list
- GET /api/patients/search
- PATCH /api/patients/update

## Error Mapping

| Condition                            | HTTP Status |
| ------------------------------------ | ----------: |
| Invalid payload                      |         400 |
| Unauthorized/Forbidden if applicable |   401 / 403 |
| Patient not found                    |         404 |
| Duplicate manual review required     |         409 |
| Unexpected controlled failure        |         500 |

## Risks

| Risk                                        | Mitigation                                              |
| ------------------------------------------- | ------------------------------------------------------- |
| Breaking existing Patient list/id contracts | Preserve existing response shape and add tests.         |
| API bypassing application layer             | Guard imports and require provider/application path.    |
| Lead replacement or dual write              | Do not import or write through Lead modules.            |
| Automated merge                             | Return review-required state only; no merge automation. |
| UI regression                               | No UI changes in this increment.                        |
