# 71.5.5 - Patient Read Services & Query Integration

## Functional Specification

**Program:** 71.5 - Controlled Development  
**Increment:** 71.5.5 - Patient Read Services & Query Integration  
**Baseline:** DENTALOPERIX_BASELINE_69_2  
**Governance Framework:** DentalOperix Governance Framework (DGF v1.0)  
**Status:** PLANNING COMPLETE / IMPLEMENTATION AUTHORIZED

---

## 1. Objective

Develop a controlled Patient read-service layer that centralizes Patient query capabilities and provides stable read DTOs for API, dashboards, administrative consumers, and future integrations.

The increment must not modify the certified command/application/persistence flow, change the Source of Truth, or introduce persistence re-architecture.

---

## 2. Functional Scope

### Included

- Patient Read Service.
- Patient read DTOs.
- Patient query contracts.
- Read adapter/mapping layer.
- Patient detail query support.
- Patient search query support.
- Patient list query support.
- Administrative profile query support.
- Refactor of eligible Patient API read consumers to use the read service.
- Architecture guards for query boundaries.
- Unit and integration tests.

### Excluded

- UI feature changes.
- Leads changes.
- Appointments changes.
- Calendar changes.
- Gmail changes.
- BookingDialog changes.
- Home changes.
- FloatingDentalAIChat changes.
- siteServices.ts changes.
- Database migrations.
- Supabase schema changes.
- Automated patient merge.

---

## 3. Target Read Architecture

```text
API / Internal Consumer
  -> Patient Read Service
  -> Patient Read Adapter / Mapper
  -> Patient Persistence Provider
  -> RelationalPatientPersistenceAdapter
  -> Supabase PostgreSQL
```

This read path is a controlled query integration and does not replace the certified command path.

---

## 4. Use Cases

### UC-71.5.5-01 - Get Patient by ID

Input: Patient ID.  
Output: PatientDetailDTO.  
Behavior: Return patient detail when found; return controlled not-found semantics otherwise.

### UC-71.5.5-02 - Search Patients

Input: query filters such as name, phone, email, document, or status.  
Output: paginated PatientSearchResultDTO collection.  
Behavior: query-only; no writes or implicit updates.

### UC-71.5.5-03 - List Patients

Input: list criteria and pagination.  
Output: PatientSummaryDTO collection.  
Behavior: stable administrative list contract.

### UC-71.5.5-04 - Administrative Profile

Input: Patient ID or functional identifier.  
Output: PatientAdministrativeProfileDTO.  
Behavior: read-only profile aggregation without lifecycle mutation.

## 5. Functional Requirements

| ID    | Requirement                                                                                 |
| ----- | ------------------------------------------------------------------------------------------- |
| RF-01 | All Patient query consumers must use Patient Read Service when in scope.                    |
| RF-02 | API routes must not contain SQL or direct relational-adapter logic.                         |
| RF-03 | Read DTOs must be specific to query contracts and must not act as domain entities.          |
| RF-04 | Read operations must not mutate persisted data.                                             |
| RF-05 | Duplicate detection must not trigger automated merge.                                       |
| RF-06 | Query errors must be mapped to controlled application/API errors.                           |
| RF-07 | Existing Patient API behavior must remain backward compatible unless explicitly documented. |

---

## 6. Non-Functional Requirements

- Maintain Ports & Adapters architecture.
- Preserve low coupling and high cohesion.
- Keep DTOs typed and stable.
- Maintain or improve test coverage.
- Avoid cross-domain dependencies.
- Preserve protected components.
- Preserve Source of Truth boundaries.

---

## 7. Acceptance Criteria

The increment is acceptable when:

- Patient Read Service is implemented and tested.
- Read DTOs are implemented and tested.
- Eligible Patient read endpoints consume Patient Read Service.
- Architecture guards prevent direct API-to-persistence read bypass where applicable.
- Full regression suite passes.
- Build and SSR validation pass where available.
- Documentation, manifest, audit, and closure evidence are updated.

---

## 8. Governance Constraints

Permanent restrictions remain active:

- No Dual Write.
- No Lead Replacement.
- No New Lead Source of Truth.
- No Persistence Re-Architecture.
- No RBAC Bypass.
- No Automated Patient Merge.

Protected components remain out of scope.
