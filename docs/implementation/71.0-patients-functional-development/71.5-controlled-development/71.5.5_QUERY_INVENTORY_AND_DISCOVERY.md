# 71.5.5 — Patient Read Services & Query Integration

## Phase 1 — Inventory and Discovery

Date: 2026-06-25
Baseline: DENTALOPERIX_BASELINE_69_2
Governance: DGF v1.0 / GML-1

## Scope

This discovery covers current Patient query access points and identifies the controlled refactor needed to centralize read behavior behind PatientReadService.

## Current Patient Query Inventory

| Area | Previous Query Boundary | 71.5.5 Determination |
|---|---|---|
| `GET /api/patients/list` | `getReadModelSource` directly from API | Refactor to PatientReadService |
| `GET /api/patients/:id` | `getReadModelSource` plus legacy administrative fallback from API | Refactor to PatientReadService |
| `GET /api/patients/search` | Patient Application Layer identity search | Refactor read query path to PatientReadService |
| Operational notifications | `listPatientAdministrativeProfiles` direct consumer call | Refactor to PatientReadService administrative profile list |
| Patient read-model provider | Aggregates worksheet/read-model data | Preserved as internal read source below PatientReadService |
| Patient admin repository | Legacy administrative profile source | Preserved below read-model fallback boundary only |

## Protected Components Review

No changes are authorized or required in:

- BookingDialog
- processDentalLead
- /api/leads/create
- Calendar
- Gmail
- FloatingDentalAIChat
- Home
- siteServices.ts

## Discovery Finding

Patient API read endpoints previously consumed read-model source/provider behavior directly. 71.5.5 introduces a single Patient Read Service boundary so external consumers no longer couple to read-model internals, diagnostics, legacy fallbacks, or persistence details.

## Governance Determination

PASS for implementation. The refactor is query-only and does not introduce persistence changes, database migrations, dual writes, Lead replacement, RBAC bypass, or automated patient merge behavior.
