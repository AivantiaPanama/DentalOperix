# 65.0 Execution Charter

## DentalOperix

Version: 1.0  
Status: OPEN  
Program Phase: 65.0 Implementation Execution

## Authorization Level

```text
IMPLEMENTATION AUTHORIZATION: APPROVED
CODE GENERATION AUTHORIZATION: SUBJECT TO PHASE-SPECIFIC WORK-PACKAGE REVIEW
DEPLOYMENT AUTHORIZATION: NOT GRANTED; REQUIRES DEPLOYMENT-SPECIFIC APPROVAL
```

## 1. Purpose

Define the execution framework for implementation activities authorized by the Program Governance Authority.

This charter establishes scope boundaries, governance controls, execution principles, approval requirements and certification criteria for 65.0.

## 2. Objectives

- Execute approved implementation work.
- Preserve certified architecture.
- Preserve certified Sources of Truth.
- Preserve governance compliance.
- Maintain rollback readiness.
- Produce certifiable implementation evidence.

## 3. Architectural Constraints

Certified architecture shall remain unchanged:

```text
Leads
-> LeadPersistencePort
-> LeadPersistenceProvider
-> RelationalLeadPersistenceAdapter
-> Supabase PostgreSQL
```

No architecture modification is authorized under this charter.

Any architecture change requires separate governance review and approval.

## 4. Source of Truth Constraints

- Leads = Source of Truth for acquisition, marketing and lead lifecycle.
- Patients = Source of Truth for person identity.
- Appointments = Source of Truth for scheduled operational events.

These boundaries remain mandatory.

## 5. Protected Components

Protected components:

- BookingDialog
- processDentalLead
- /api/leads/create
- Calendar
- Gmail
- FloatingDentalAIChat
- Home
- siteServices.ts

Default rule:

```text
NO MODIFICATION
```

Any proposed impact requires:

- Architecture Review
- Governance Review
- Explicit Approval

## 6. In Scope

65.0 may execute only work that:

- originates from certified designs
- complies with 57.x-64.x certifications
- complies with governance restrictions
- passes architectural review
- is documented in an approved work package

## 7. Out of Scope

Not authorized:

- Persistence Re-Architecture
- New Source of Truth creation
- Lead Replacement
- Dual Write
- RBAC Bypass
- Automated Patient Merge

Not authorized without separate approval:

- Protected component modification
- Architectural redesign
- Production deployment

## 8. Execution Model

Execution shall proceed through controlled work packages.

Each work package requires:

1. Architectural Review
2. Dependency Review
3. Risk Review
4. Technical Impact Review
5. Governance Determination
6. Explicit Approval

Only after approval may implementation activities begin.

## 9. Validation Requirements

Every work package shall include:

- Architectural validation
- Governance validation
- Regression validation
- Evidence collection
- Certification review

## 10. Exit Criteria

65.0 may be closed only when:

- Approved scope completed
- Validation completed
- Evidence collected
- Certification completed
- Governance retrospective completed
