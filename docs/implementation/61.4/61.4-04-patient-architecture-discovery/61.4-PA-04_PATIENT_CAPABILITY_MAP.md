# 61.4-PA-04 Patient Capability Map

Phase: 61.4  
Workstream: 61.4-04 Patient Architecture Discovery  
Status: DISCOVERY  
Implementation: NOT AUTHORIZED  
Code Generation: NOT AUTHORIZED  
Schema Changes: NOT AUTHORIZED

---

## 1. Purpose

Classify Patient-domain capabilities into candidate categories so future phases can prioritize design safely.

This document does not authorize implementation.

---

## 2. Capability Classification

Capabilities are classified as:

- Core Candidate
- Governance Candidate
- Future Candidate
- Out of Scope
- Prohibited

---

## 3. Core Candidate Capabilities

| Capability | Description | Notes |
| --- | --- | --- |
| Patient Identity Management | Maintain durable person identity. | Central Patient responsibility. |
| Patient Contact Point Management | Manage emails, phones, and preferred contact context. | Signals, not absolute identity proof. |
| Patient Lifecycle State | Active, archived, under review, etc. | Requires governance. |
| Patient Cross-Domain Association Tracking | Track Lead and Appointment associations. | Must preserve SoT boundaries. |

---

## 4. Governance Candidate Capabilities

| Capability | Description | Notes |
| --- | --- | --- |
| Identity Change Audit | Audit identity changes. | Required. |
| Duplicate Candidate Detection | Detect possible duplicates. | Detection only. |
| Duplicate Review Workflow | Human review of possible duplicates. | No automated merge. |
| Identity Correction Governance | Govern sensitive corrections. | Requires RBAC alignment. |
| Association Review Governance | Govern Lead/Appointment associations. | Prevent ownership transfer. |

---

## 5. Future Candidate Capabilities

| Capability | Description | Notes |
| --- | --- | --- |
| Multi-Clinic Patient Identity | Identity across clinic boundaries. | Future discovery required. |
| Patient Consent Model | Consent for communication / data processing. | Future compliance workstream. |
| Clinical Record Relationship Model | Relationship to future clinical records. | Out of current architecture decision. |
| Patient Portal Identity | Patient-facing identity. | Future product capability. |
| Advanced Identity Scoring | Sophisticated matching signals. | Must remain human-reviewed. |

---

## 6. Out of Scope for 61.4 Discovery

The following should not be designed or implemented during 61.4 discovery:

- database schema
- migration scripts
- API endpoints
- UI components
- merge services
- direct Supabase writes
- clinical record architecture
- patient portal architecture

---

## 7. Prohibited Capabilities

| Capability | Reason |
| --- | --- |
| Automated Patient Merge | Explicitly prohibited. |
| Lead Replacement | Explicitly prohibited. |
| New Lead Source of Truth | Violates 57.x certified architecture. |
| Persistence Re-Architecture | Violates certified baseline. |
| RBAC Bypass | Explicitly prohibited. |
| Appointment-Owned Identity | Violates Appointments SoT boundary. |

---

## 8. Prioritization Candidate

The safest future design sequence is:

1. Patient Identity Management
2. Identity Change Audit
3. Contact Point Management
4. Duplicate Candidate Detection
5. Duplicate Review Workflow
6. Cross-Domain Association Tracking

This is not an implementation plan. It is a discovery-level sequencing candidate.

---

## 9. Discovery Conclusion

The Patient capability map supports a phased, governance-first evolution of the Patient domain.

The next safe step after architecture discovery should be closure review or a controlled design-readiness assessment, not implementation.

Implementation remains NOT AUTHORIZED.
