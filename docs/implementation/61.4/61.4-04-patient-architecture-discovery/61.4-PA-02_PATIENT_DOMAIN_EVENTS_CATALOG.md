# 61.4-PA-02 Patient Domain Events Catalog

Phase: 61.4  
Workstream: 61.4-04 Patient Architecture Discovery  
Status: DISCOVERY  
Implementation: NOT AUTHORIZED  
Code Generation: NOT AUTHORIZED  
Schema Changes: NOT AUTHORIZED

---

## 1. Purpose

Define a conceptual catalog of Patient domain events to support future architecture, governance, auditability, and traceability discussions.

This catalog is not an implementation contract.

---

## 2. Event Design Principle

Patient events should describe meaningful business facts about person identity.

They must not trigger or imply:

- automated patient merge
- lead replacement
- appointment ownership transfer
- persistence re-architecture
- RBAC bypass

---

## 3. Candidate Event Categories

### Identity Lifecycle Events

| Event | Meaning | Governance Level |
| --- | --- | --- |
| PATIENT_CREATED | A Patient identity was created. | Required audit |
| PATIENT_ACTIVATED | A Patient identity became active. | Required audit |
| PATIENT_ARCHIVED | A Patient identity was archived. | Required audit |
| PATIENT_REACTIVATED | A Patient identity was restored from archived state. | Required audit |

---

### Identity Attribute Events

| Event | Meaning | Governance Level |
| --- | --- | --- |
| PATIENT_IDENTITY_UPDATED | Identity data changed. | Required audit |
| PATIENT_NAME_CORRECTED | Name was corrected. | Required audit |
| PATIENT_BIRTH_DATE_CORRECTED | Birth date was corrected. | Enhanced audit |
| PATIENT_CONTACT_POINT_ADDED | Contact point was added. | Required audit |
| PATIENT_CONTACT_POINT_UPDATED | Contact point was updated. | Required audit |
| PATIENT_CONTACT_POINT_REMOVED | Contact point was removed. | Required audit |
| PATIENT_PRIMARY_CONTACT_CHANGED | Primary contact changed. | Enhanced audit |

---

### Duplicate Detection and Review Events

| Event | Meaning | Governance Level |
| --- | --- | --- |
| PATIENT_DUPLICATE_CANDIDATE_DETECTED | Possible duplicate detected. | Required audit |
| PATIENT_DUPLICATE_REVIEW_STARTED | Human review started. | Required audit |
| PATIENT_DUPLICATE_REVIEW_DISMISSED | Candidate dismissed. | Required audit |
| PATIENT_DUPLICATE_REVIEW_CONFIRMED | Candidate confirmed as possible duplicate. | Enhanced audit |
| PATIENT_DUPLICATE_REVIEW_ESCALATED | Review escalated. | Enhanced audit |

Important: PATIENT_DUPLICATE_REVIEW_CONFIRMED does not authorize automated merge.

---

### Cross-Domain Association Events

| Event | Meaning | Governance Level |
| --- | --- | --- |
| PATIENT_LEAD_ASSOCIATION_PROPOSED | Lead may relate to Patient. | Required audit |
| PATIENT_LEAD_ASSOCIATION_CONFIRMED | Lead association confirmed. | Required audit |
| PATIENT_LEAD_ASSOCIATION_REJECTED | Lead association rejected. | Required audit |
| PATIENT_APPOINTMENT_ASSOCIATION_PROPOSED | Appointment may relate to Patient. | Required audit |
| PATIENT_APPOINTMENT_ASSOCIATION_CONFIRMED | Appointment association confirmed. | Required audit |
| PATIENT_APPOINTMENT_ASSOCIATION_REJECTED | Appointment association rejected. | Required audit |

Associations must preserve source-of-truth boundaries.

---

## 4. Explicitly Disallowed Events

The following events must not be introduced under current governance:

| Event | Reason |
| --- | --- |
| PATIENT_AUTO_MERGED | Automated Patient Merge is prohibited. |
| LEAD_REPLACED_BY_PATIENT | Lead Replacement is prohibited. |
| APPOINTMENT_IDENTITY_OWNERSHIP_TRANSFERRED | Appointments are not source of truth for identity. |
| PATIENT_DIRECT_SUPABASE_WRITTEN | Persistence re-architecture / bypass risk. |

---

## 5. Event Audit Requirements

Future implementation proposals must define at minimum:

- actor
- timestamp
- event type
- affected patient identity
- before/after values where applicable
- reason or review outcome where applicable
- source context where applicable

---

## 6. Discovery Conclusion

Patient domain events should support traceability and governance first.

They must not become a mechanism for automated identity consolidation or cross-domain ownership transfer.

Implementation remains NOT AUTHORIZED.
