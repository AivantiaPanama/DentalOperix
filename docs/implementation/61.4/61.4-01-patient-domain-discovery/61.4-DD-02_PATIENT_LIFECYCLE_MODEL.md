## 2. Lifecycle Principle

A Patient represents a permanent person identity.

The Patient normally must not be deleted.

The Patient changes state as operational circumstances change.

Approved states:

```text
active
inactive
lost_contact
archived
```

---

## 3. Lifecycle States

### active

Meaning:

The Patient is operationally active or currently eligible for clinic services.

Examples:

- newly created patient
- patient with recent activity
- patient with upcoming appointment
- patient actively communicating with clinic

---

### inactive

Meaning:

The Patient is known by the clinic but has no recent active engagement.

Examples:

- no recent appointment
- no active treatment flow
- reachable but not currently active

---

### lost_contact

Meaning:

The Patient exists, but the clinic cannot currently reach them through available contact points.

Examples:

- failed calls
- invalid email
- WhatsApp unreachable
- repeated no-response attempts

---

### archived

Meaning:

The Patient is retained for administrative or historical continuity but is no longer part of active operational workflows.

Examples:

- administrative archive
- historical-only retention
- long-term inactive state after governance-approved criteria

---

## 4. Patient Creation Sources

Approved creation sources:

```text
web
chat
whatsapp
phone
walk_in
lead
appointment
assistant
admin
doctor
```

Creation from any source must respect identity resolution before new Patient creation.

Patient creation from Lead does not replace Lead.

Patient creation from Appointment does not replace Appointment.

## 7. State Transition Candidate Matrix

| From | To | Allowed? | Governance Note |
|---|---:|---:|---|
| none | active | Yes | Default creation candidate |
| active | inactive | Yes | Requires business criteria |
| active | lost_contact | Yes | Requires failed contact criteria |
| active | archived | Yes | Requires explicit administrative reason |
| inactive | active | Yes | Patient re-engages |
| inactive | lost_contact | Yes | Contact attempts fail |
| inactive | archived | Yes | Requires archive policy |
| lost_contact | active | Yes | Contact restored |
| lost_contact | inactive | Yes | Contact restored but no active workflow |
| lost_contact | archived | Yes | Requires archive policy |
| archived | active | Conditional | Requires governance review/audit |
| archived | inactive | Conditional | Requires governance review/audit |
| any | deleted | No | Deletion is not normal workflow |

---

## 8. Lifecycle Guardrails

1. Patient identity must persist across contact changes.
2. Lifecycle state must not imply Lead lifecycle state.
3. Lifecycle state must not imply Appointment state.
4. Archived does not mean physically deleted.
5. Lost contact must not trigger automatic merge or replacement.
6. Manual merge must be auditable and authorized.
7. Automated merge remains prohibited.

---

## 9. Audit Considerations

Lifecycle changes that should be auditable:

- creation
- lifecycle state change
- identity-significant attribute change
- contact point deactivation
- duplicate candidate marking
- manual merge request
- manual merge completion
- archive/reactivation

Audit implementation is not authorized by this document.
