## 1. Executive Summary

DentalOperix currently has certified foundations for:

- Program 57.x Persistence Transition: CLOSED / CERTIFIED
- Production Cutover: CERTIFIED
- 61.0 Documentation Governance: COMPLETE
- 61.1 Users + Authentication + RBAC + Dashboard Routing: CLOSED / CERTIFIED
- 61.2 Assistant / Front Desk Workspace: IN PROGRESS

The current 61.2 workspace supports front-desk operations around leads, including:

- Assistant Workspace Shell
- Today's Schedule
- Lead Queue
- Lead Detail Read-Only
- Lead Status Management
- Lead Notes

However, a clinic requires a minimum operational foundation to function day to day:

1. Register internal staff users such as administrators, doctors, and assistants.
2. Assign roles and account status to those users.
3. Register patients who arrive directly at the clinic for consultation.
4. View, search, and maintain basic patient administrative records.

This document establishes 61.3 as the recommended next major operational foundation after 61.2 closure.

## 6. Problem Statement

The product has strong certified foundations for lead handling and assistant/front-desk workflow, but it does not yet provide a complete minimum operating model for a clinic.

Operational gap examples:

- A clinic administrator cannot yet complete a certified UI flow to create a doctor account and assign credentials.
- A clinic administrator cannot yet complete a certified UI flow to create an assistant account and assign role/status.
- A front desk assistant cannot yet complete a certified flow to register a walk-in patient as a patient administrative record.
- A doctor cannot yet rely on a certified patient directory as a baseline operational record.

Without these flows, continued feature development risks building advanced modules before the clinic can perform basic day-to-day setup and intake.

---

## 7. Product Goal

Define and implement a minimum viable clinic operating foundation that allows the clinic to operate without interrupting ongoing development.

The goal is not to build a full EHR, full patient lifecycle, treatment plan system, or doctor-patient assignment system.

The goal is to create a certified operational baseline:

```text
Administrator can manage staff users.
Front desk can register and maintain basic patient administrative records.
The system remains compliant with certified Leads architecture.
```

## 9. PR-61.3-01 — User Administration Foundation

### Objective

Allow an administrator to create and maintain internal clinic users.

### User Types

Supported roles must remain aligned with 61.1 RBAC:

```text
administrator
doctor
assistant
patient
```

For 61.3-01, focus should be internal staff users:

- administrator
- doctor
- assistant

Patient users should not be conflated with patient administrative records unless explicitly approved by Architecture Review.

### Minimum Capabilities

- Create staff user
- Assign role
- Set display name
- Set email
- Set account status
- Activate/deactivate user
- Prevent unauthorized role assignment

### Credential Handling

Credential strategy must be reviewed before implementation.

Allowed options to evaluate:

1. Admin creates user with temporary password.
2. Admin creates user in pending activation state.
3. Admin triggers password setup/reset process.

No implementation should hard-code passwords, bypass authentication, or weaken RBAC.

## 10. PR-61.3-02 — Patient Administration Foundation

### Objective

Allow front desk/admin users to create and maintain basic administrative patient records for patients who arrive directly at the clinic.

### Important Boundary

This is a patient administrative record, not a certified Lead ↔ Patient relationship model.

61.3-02 must not assume:

```text
Lead -> Patient conversion
```

and must not assume:

```text
Doctor -> Patient assignment
```

unless Architecture Review explicitly approves those models.

### Minimum Patient Fields

Candidate fields for review:

- First name
- Last name
- Phone
- Email
- Date of birth
- Address
- Emergency contact name
- Emergency contact phone
- Preferred contact method
- Administrative notes
- Status

### Minimum Capabilities

- Create patient administrative record
- Edit patient administrative record
- View patient administrative record
- Basic validation for required fields
- No lead mutation unless explicitly approved

---

## 11. PR-61.3-03 — Patient Directory

### Objective

Allow authorized clinic staff to find and view patient administrative records.

### Minimum Capabilities

- Patient list
- Basic search by name, phone, or email
- Patient detail view
- Edit entry point for authorized roles

### Explicitly Out of Scope

- Global search across all entities
- Real-time patient updates
- Treatment history
- Medical charting
- Doctor assignment
- Lead-patient merge
- Duplicate resolution automation
