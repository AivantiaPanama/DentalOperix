# DentalOperix 61.3-01 — Patient Identity Foundation Opening

**Status:** OPEN / ARCHITECTURE REVIEW REQUIRED BEFORE IMPLEMENTATION  
**Date:** 2026-06-23  
**Predecessor:** 61.3-00 Patient Domain Discovery COMPLETE / APPROVED

---

## 1. Objective

Build the technical foundation for the Patient identity domain before implementing patient workspace UI.

61.3-01 must create reusable domain, service, repository and persistence foundations for Patient identity.

---

## 2. Scope

Included:

- Patient domain model.
- Patient lifecycle states.
- Patient contact point contracts.
- Patient identifier contracts.
- Basic retired/insurance administrative flags or contracts.
- Patient validation.
- Patient repository interface.
- Patient application service.
- Identity resolution skeleton.
- Manual merge contract skeleton.
- Audit event contract skeleton.
- Normalized SQL migration proposal.
- Tests for domain, validation, identity resolution and repository mapping.

Excluded:

- Patient Workspace UI.
- Patient Directory UI.
- Doctor ↔ Patient assignment.
- Clinical records.
- Treatment plans.
- Treatment Episode implementation.
- Billing automation.
- Insurance coverage calculations.
- Marketing consent automation.
- Multi-clinic tenancy.
- Automated patient merge.

---

## 3. Architecture Constraints

Use reusable domain architecture:

```text
Domain
  -> Application Service
  -> Repository / Adapter
  -> Infrastructure
  -> UI
```

Do not place identity, duplicate, merge, insurance or lifecycle business rules inside UI components.
