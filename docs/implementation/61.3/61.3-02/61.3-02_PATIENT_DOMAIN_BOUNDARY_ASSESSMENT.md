## 1. Executive Summary

This assessment inventories Patient-related artifacts in the project and classifies their ownership relative to the 61.3-01 Patient Identity Foundation.

### Core Finding

There are two categories of Patient artifacts:

1. **Official 61.3 Patient Identity Foundation** under `src/server/patients/*`.
2. **Pre-existing Patient Management / Read Model / Administrative Profile artifacts** under `src/routes/api/patients/*`, `src/lib/patients/*`, and `src/server/read-models/patient-*`.

The official 61.3 Patient Identity Foundation is properly isolated and should be treated as the authoritative domain implementation for Patient identity going forward.

The historical Patient Management artifacts remain active for current read/admin workflows and should be classified as **Legacy / Read Model / Administrative Overlay**, not as the official Patient identity domain.

## 5. Conflict Analysis

### Conflict 1 — Naming Collision

There is naming overlap between the new official Patient Foundation and older Patient Management artifacts.

**Examples:**

```text
src/server/patients/patient-domain.ts
src/server/patients/admin-repository.ts
src/lib/patients/admin-profile.ts
src/routes/api/patients/*
```

**Impact:** Medium.

**Assessment:** Not a current functional conflict, but it can confuse future implementation ownership.

**Required Control:** Document that only the 61.3 Patient Foundation owns Patient identity domain and relational persistence.

---

### Conflict 2 — Legacy Administrative Overlay Writes

`src/server/patients/admin-repository.ts` writes to:

```text
.data/patient-administrative-profiles.json
```

This is not the official 61.3 Patient identity store.

**Impact:** Medium.

**Assessment:** This is an existing administrative overlay, not a Patient identity source of truth. It must not be expanded into a second Patient source of truth.

**Required Control:** Before Patient UI integration, decide whether this overlay remains legacy, is migrated, or is retired.

---

### Conflict 3 — Lead-derived Patient Administrative Profiles

`admin-repository.ts` derives base profiles from:

```text
leadPersistenceProvider.getActiveLeadPersistenceAdapter().listLeads()
```

**Impact:** Medium.

**Assessment:** This is acceptable only as a legacy read/admin projection. It must not become Patient identity persistence and must not mutate Leads.

**Governance Note:** Leads remain Source of Truth for acquisition and commercial follow-up. Patient remains identity. This derivation does not replace Lead as long as it remains read/projection behavior.

---

### Conflict 4 — UI Exists Before New Patient Domain Integration

Existing Patient UI artifacts exist, but 61.3-02 excludes UI.

**Impact:** Medium.

**Assessment:** No action in 61.3-02. Future UI work requires explicit architecture review.

## 8. Findings

### H-61.3-02-002 — Legacy Patient Boundary Requires Formal Classification

**Status:** OPEN / CONTROLLED
**Severity:** Medium
**Area:** Domain ownership clarity

Existing Patient Management APIs, read models and administrative overlays remain active. They do not currently invalidate 61.3 Patient Foundation, but they require clear classification before any Patient UI or operational integration work.

**Recommendation:** Create a follow-up alignment decision before Patient UI begins:

```text
Legacy Patient Management Alignment Decision
```

This decision should classify each legacy artifact as:

```text
Retain as read model
Adapt to official Patient Foundation
Deprecate
Migrate
```

---

### H-61.3-02-004 — Administrative Overlay Store Is Not Official Patient Persistence

**Status:** OPEN / CONTROLLED
**Severity:** Medium
**Area:** Data ownership

The file-backed store:

```text
.data/patient-administrative-profiles.json
```

is used by legacy administrative profile flows. It must not be considered official Patient identity persistence.

**Recommendation:** Prevent any new Patient Foundation integration from depending on this store as a source of truth.
