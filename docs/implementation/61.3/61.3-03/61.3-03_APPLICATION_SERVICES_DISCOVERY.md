# 61.3-03_APPLICATION_SERVICES_DISCOVERY

Project: DentalOperix  
Phase: 61.3-03 Patient Application Services  
Baseline: DENTALOPERIX_BASELINE_61_3_DISCOVERY_COMPLETE

Status: DISCOVERY COMPLETE

---

## 1. Objective

Define the Application Services layer that will orchestrate Patient Domain operations without introducing UI concerns, persistence ownership changes, or violations of the certified architecture.

This phase establishes:

```text
Application Service Inventory
Service Responsibilities
Service Boundaries
Interaction Patterns
Dependency Rules
Validation Criteria
```

No UI implementation is authorized in this phase.

---

## 2. Architectural Context

Certified architecture at the start of 61.3-03:

```text
Patient Domain
↓
Patient Application Services
↓
Patient Repository
↓
Relational Repository
↓
Supabase PostgreSQL
```

Patient Domain remains authoritative for:

```text
Identity Rules
Lifecycle Rules
Merge Rules
Audit Rules
```

Application Services remain orchestration-only.

---

## 3. Governance Constraints

The following rule remains mandatory:

```text
Leads = Source of Truth
```

for:

```text
Lead Capture
Marketing
Commercial Tracking
Conversion Tracking
```

Patient Domain does not replace Lead Domain.

The following remain prohibited:

```text
Dual Write
Lead Replacement
Secondary Source of Truth
Persistence Re-Architecture
RBAC Bypass
Analytics Write Back
```

Protected components remain out of scope:

```text
BookingDialog
processDentalLead
/api/leads/create
Calendar
Gmail
FloatingDentalAIChat
Home
siteServices.ts
```

---

## 4. Service Inventory

Approved for discovery:

```text
PatientCreationService
PatientUpdateService
PatientArchiveService
PatientIdentityResolutionService
PatientMergePreparationService
PatientAuditService
```

---

## 5. Application Service Boundaries

Application Services may:

```text
Call Domain Logic
Call Repositories
Call Audit Services
Coordinate Workflows
```

Application Services may not:

```text
Contain Core Business Rules
Perform Direct SQL
Perform UI Logic
Bypass Repositories
Modify Lead Authority
```

---

## 6. Discovery Conclusion

The Application Services layer is approved as the next architectural layer above the certified Patient Foundation.

Patient Domain remains authoritative.

Repositories remain authoritative for persistence access.

Application Services remain orchestration-only.

No UI implementation is authorized during this phase.
