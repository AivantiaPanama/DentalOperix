---

## 1. Purpose

Define the architectural structure of the Patient Application Services layer.

This layer exists to:

```text
Coordinate workflows
Invoke Domain Rules
Invoke Repositories
Invoke Audit Contracts
```

This layer does not:

```text
Own Business Rules
Own Persistence
Own UI Logic
```

---

## 2. Certified Architectural Position

```text
Patient Domain
        ↓
Patient Application Services
        ↓
Patient Repository Contracts
        ↓
Relational Patient Repository
        ↓
Supabase PostgreSQL
```

Authority remains:

```text
Patient Domain
```

Application Services remain orchestration-only.

---

## 3. Architectural Principles

### AP-61.3-03-001 Domain Authority Principle

Business rules belong exclusively to Patient Domain.

### AP-61.3-03-002 Repository Isolation Principle

Application Services interact only through PatientRepository contracts.

Forbidden:

```text
Direct SQL
Direct Supabase Calls
Persistence Bypass
```

### AP-61.3-03-003 UI Independence Principle

Application Services must not depend on React components, pages, routes, or UI state.

### AP-61.3-03-004 Lead Boundary Protection

Application Services must not modify Lead persistence, Lead lifecycle, or Lead authority.
