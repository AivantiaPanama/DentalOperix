---

## 1. Purpose

Define authoritative ownership, responsibilities, dependencies and implementation boundaries for the Patient Domain architecture.

Primary objective:

```text
Prevent Responsibility Drift
Prevent Layer Leakage
Protect Certified Architecture
```

---

## 2. Layer Ownership Matrix

| Layer | Authority | Responsibility | Owns Business Rules | Owns Persistence | Owns UI |
|---|---|---|---|---|---|
| Domain | YES | Patient Identity | YES | NO | NO |
| Application Services | NO | Workflow Orchestration | NO | NO | NO |
| Repository Contracts | NO | Persistence Abstraction | NO | NO | NO |
| Repository Implementation | NO | Persistence Access | NO | YES | NO |
| Infrastructure | NO | Storage Technology | NO | YES | NO |
| UI (Future) | NO | User Interaction | NO | NO | YES |

## 5. Lead Boundary Matrix

Leads remain Source of Truth for:

```text
Lead Capture
Lead Tracking
Lead Conversion
Commercial Funnel
Marketing Attribution
```

Patient Services may not:

```text
Modify Lead Persistence
Replace Lead Authority
Create Secondary Lead State
```

## 7. Final Boundary Resolution

```text
Patient Domain = Architectural Authority
Application Services = Workflow Authority
Repositories = Persistence Authority
Infrastructure = Storage Authority
```

No boundary conflicts were identified.
