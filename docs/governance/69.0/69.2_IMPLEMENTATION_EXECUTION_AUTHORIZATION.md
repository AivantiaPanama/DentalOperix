# DENTALOPERIX 69.2 — Implementation Execution Authorization

**Status:** CLOSED / IMPLEMENTATION EXECUTION AUTHORIZED  
**Governance Mode:** Architect Principal · Technical Reviewer · Governance Guardian  
**Baseline:** DENTALOPERIX_BASELINE_69_2

## 1. Objetivo

Autorizar el paso de **Implementation Planning** a **Implementation Execution** para el dominio **Patients**, sujeto a las restricciones certificadas.

## 2. Alcance autorizado

La autorización permite cambios únicamente dentro del dominio Patients:

- Patient Domain
- PatientPersistencePort
- PatientPersistenceProvider
- RelationalPatientPersistenceAdapter
- Patient Repository
- Patient Services
- Artefactos propios del dominio Patients

## 3. Análisis arquitectónico

Leads:

```text
LeadPersistencePort
→ LeadPersistenceProvider
→ RelationalLeadPersistenceAdapter
→ Supabase PostgreSQL
```

Patients:

```text
PatientPersistencePort
→ PatientPersistenceProvider
→ RelationalPatientPersistenceAdapter
→ Supabase PostgreSQL
```

Resultado:

- No re-arquitectura.
- No nuevo Source of Truth.
- No sustitución de Leads.
- Separación de dominios preservada.
- Port → Provider → Adapter obligatorio.

## 4. Componentes protegidos fuera del alcance

- BookingDialog
- processDentalLead
- /api/leads/create
- Calendar
- Gmail
- FloatingDentalAIChat
- Home
- siteServices.ts

## 5. Restricciones permanentes

| Restricción | Estado |
|---|---|
| No Dual Write | Vigente |
| No Lead Replacement | Vigente |
| No New Lead Source of Truth | Vigente |
| No Persistence Re-Architecture | Vigente |
| No RBAC Bypass | Vigente |
| No Automated Patient Merge | Vigente |
| Componentes protegidos sin cambios | Vigente |

## 6. Impacto técnico

Impacto permitido: implementación controlada del dominio Patients únicamente.

Impacto no permitido: modificaciones en Leads, Marketing, Booking, Calendar, Gmail, AI o componentes protegidos.

## 7. Compatibilidad 57.x–69.1

Compatible con todas las certificaciones vigentes hasta 69.1.

## 8. Condiciones de autorización

Durante ejecución se debe:

- Respetar Port → Provider → Adapter.
- No modificar componentes protegidos.
- No introducir Dual Write.
- No alterar Sources of Truth.
- Solicitar nueva revisión si hay desviación arquitectónica.
- Generar validación y evidencia por incremento.
