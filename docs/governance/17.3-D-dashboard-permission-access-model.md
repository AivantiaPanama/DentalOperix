# 17.3-D — Dashboard Permission & Access Model Design

## Estado

IMPLEMENTED AS GOVERNED ACCESS MODEL CONTRACTS

## Alcance

Esta fase define el modelo de permisos y acceso futuro para Executive Dashboard UI sin implementar UI funcional, rutas, login, almacenamiento de credenciales ni APIs nuevas.

## Análisis arquitectónico

La capa de acceso queda modelada como un gate declarativo entre una futura UI y los contratos internos ya aprobados:

```text
Future Dashboard UI
        ↓
Dashboard Permission & Access Model
        ↓
ExecutiveDashboardApiContracts
        ↓
Executive Dashboard API
```

El acceso autorizado exige exclusivamente el permiso:

```text
executive-observability:read
```

El modelo no concede acceso a aggregates, adapters, read sources, read models directos, telemetría cruda ni paths de escritura.

## Riesgos

1. Acoplamiento accidental con `/admin/login`.
   - Mitigación: esta fase no implementa login ni almacenamiento de credenciales.

2. Expansión de permisos hacia datos operativos.
   - Mitigación: el único permiso permitido es `executive-observability:read`.

3. Exposición de telemetría cruda.
   - Mitigación: políticas `metric-only-read` y pruebas contra eventos crudos.

4. Acceso directo a aggregates o adapters.
   - Mitigación: dependencias prohibidas declaradas y validadas por pruebas.

## Resultado formal

17.3-D — Dashboard Permission & Access Model Design queda implementado como contratos y pruebas de gobernanza.

Validaciones:

- Executive Dashboard, Operational Dashboard y Governance Dashboard tienen políticas separadas.
- Todas las políticas usan `executive-observability:read`.
- Todas las políticas son `metric-only-read`.
- No existe implementación UI.
- No existe acceso directo a aggregates.
- No existe acceso directo a adapters.
- No existe exposición de raw telemetry.
- No existe modificación de login.
- No existe almacenamiento de credenciales.
- Leads permanece Source of Truth.
- READ MODEL PLATFORM v2 permanece CLOSED/FROZEN.
