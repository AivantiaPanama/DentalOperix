# 61.0 Multi-AI Operating Model

Status: ACTIVE
Owner: Product Governance / AI Delivery Governance
Last updated: 2026-06-20
Project: DentalOperix

---

## 1. Purpose

This document defines the official operating model for using multiple AI systems inside the DentalOperix Software Factory.

The purpose is to accelerate delivery while preserving:

- Certified architecture
- Product governance
- Technical consistency
- Documentation continuity
- Release readiness
- `Leads = Source of Truth`

No AI system is authorized to modify certified architecture without explicit architectural review and approval.

---

## 2. Strategic Objective

DentalOperix uses a coordinated multi-AI model to reduce delivery time, improve quality, and separate responsibilities across specialized AI tools.

The official certified architecture remains:

```text
Leads
→ LeadPersistencePort
→ LeadPersistenceProvider
→ RelationalLeadPersistenceAdapter
```

Permanent rule:

```text
Leads = Source of Truth
```

- Unauthorized architecture changes

## 5. AI Roles and Responsibilities

### 5.1 ChatGPT — Architect Principal / Governance Guardian / AI Delivery Coordinator

ChatGPT is responsible for:

- Architecture review
- Technical governance
- Dependency analysis
- Risk analysis
- Product roadmap alignment
- Release readiness review
- Final certification review
- Integration of outputs from other AI tools
- Validation against certified architecture

ChatGPT must approve architectural direction before implementation begins.

ChatGPT must not generate implementation code until explicit approval is given.

### 5.2 Claude — Functional Documentation Specialist

Claude is responsible for:

- User stories
- Acceptance criteria
- Business rules
- Role definitions
- Permission matrices
- Functional specifications
- Product documentation drafts
- Workflow descriptions

Claude must not define architecture independently.

Claude outputs must be reviewed by ChatGPT before being passed to Cursor.

### 5.3 Cursor — Implementation Agent

Cursor is responsible for:

- Controlled implementation
- Refactoring
- Unit tests
- Integration tests
- Pull request preparation
- Codebase alignment
- Technical task execution

Cursor must follow approved specifications only.

Cursor must not:

- Change certified architecture
- Introduce new persistence patterns
- Modify protected components without approval
- Create new sources of truth
- Bypass RBAC
- Perform broad refactors outside the task scope

### 5.4 v0 / Figma — UX and Interface Design

v0 / Figma are responsible for:

- Wireframes
- Dashboard layouts
- UI component proposals
- UX flows
- Visual structure
- Frontend mockups

v0 / Figma outputs are design proposals only.

They must not define backend behavior, database structure, persistence logic, or authorization rules.

## 7. Handoff Rules

### 7.1 ChatGPT to Claude

ChatGPT provides:

- Architectural boundaries
- Product objective
- Governance constraints
- Required deliverables
- Open questions
- Non-negotiable rules

Claude returns:

- Functional specification
- User stories
- Acceptance criteria
- Business rules
- Role and permission definitions

### 7.2 Claude to ChatGPT

ChatGPT reviews Claude outputs for:

- Architecture compatibility
- Product consistency
- RBAC correctness
- Commercial alignment
- Missing edge cases
- Conflicts with certified rules

Only approved outputs move forward.

### 7.3 ChatGPT to v0 / Figma

ChatGPT provides:

- Approved functional scope
- Role-based UX requirements
- Dashboard purpose
- Required screens
- Navigation constraints
- Protected areas

v0 / Figma returns:

- Wireframes
- Component layouts
- Dashboard structures
- UX recommendations

### 7.4 ChatGPT to Cursor

ChatGPT provides:

- Approved implementation plan
- Affected files
- Forbidden files
- Architecture constraints
- Test requirements
- Acceptance criteria

Cursor returns:

- Code changes
- Tests
- Pull request summary
- Implementation notes
- Known limitations

### 7.5 Cursor to ChatGPT

ChatGPT performs:

- Architecture review
- Technical review
- Governance review
- Test review
- Documentation update validation
- Release readiness assessment

No merge should occur before this review is complete.

- Tests pass

- RBAC rules are enforced where applicable
- Documentation is updated
- Pull request is reviewed
- Release readiness impact is documented

- Unauthorized changes to protected files

- Introduction of new source of truth
- Dual write patterns
- Lead replacement
- Analytics write back

### Product Risks

- Building dashboards before RBAC
- Building patient management before user identity
- Incomplete commercial readiness

### Continuity Risks

- AI outputs not documented
- Decisions not recorded
- Implementation not linked to roadmap
