# 17.3-E — Dashboard Data Client Design

## Status

APPROVED FOR FUTURE DASHBOARD CLIENT IMPLEMENTATION

## Scope

This phase defines the governed dashboard data client design only. It does not implement UI, visual components, routes, API handlers, fetch transport, persistence, browser storage, credential forwarding, domain logic, fallback logic, or client-side aggregation.

## Architectural rule

The future dashboard data client may only prepare metric-only request descriptors against `ExecutiveDashboardApiContracts`, gated by the 17.3-D access model and the `executive-observability:read` permission.

Allowed path:

```text
Dashboard Data Client Design
  -> ExecutiveDashboardAccessModel
  -> ExecutiveDashboardApiContracts
  -> Executive Dashboard API
```

Forbidden paths:

```text
Data Client -> Aggregates
Data Client -> Adapters
Data Client -> Raw Telemetry
Data Client -> Read Sources
Data Client -> ReadModelSourceProvider
Data Client -> Lead write paths
Data Client -> Browser credential storage
Data Client -> Client-side aggregation
```

## Implemented artifacts

- `src/server/read-models/executive-dashboard-data-client-design.ts`
- `src/server/read-models/executive-dashboard-data-client-design.test.ts`

## Endpoint bindings

The design binds exactly three dashboard surfaces:

- Executive Dashboard -> `/api/internal/executive-observability/executive`
- Operational Dashboard -> `/api/internal/executive-observability/operational`
- Governance Dashboard -> `/api/internal/executive-observability/governance`

All bindings are:

- `GET`
- `metric-only`
- `ExecutiveDashboardApiContracts` only
- `transport-contract-only`
- `projection-free-display-mapping-only`
- permissioned by `executive-observability:read`
