# 15.5-A.6 — Patient Management Read Model Pilot Validation

## Objective

Validate the Patient Management pilot after migrating read-only patient list and detail endpoints to the read model source path.

## Validated read flow

```text
PatientManagementWorkspace
  -> GET /api/patients/list
  -> GET /api/patients/:id
  -> ReadModelSourceProvider
  -> PatientAggregateReadService
  -> Patient read adapters
  -> Read models
  -> Leads fallback
```

## Contract compatibility

The public API contracts remain unchanged.

### GET /api/patients/list

```ts
{
  success: boolean;
  patients: PatientAdministrativeProfile[];
  error?: string;
}
```

### GET /api/patients/:id

```ts
{
  success: boolean;
  patient: PatientAdministrativeProfile;
  error?: string;
}
```

Internal read-model diagnostics, including resolved identity metadata, are not exposed through either response.

## Write path status

The following write paths remain unchanged and continue to use the existing Leads-backed administrative repository:

- PATCH /api/patients/:id/admin-profile
- POST /api/patients/:id/verify-profile

No dual write, production migration, or replacement of Leads was introduced.

## Restricted files

This phase does not require changes to restricted components or integration surfaces:

- BookingDialog
- processDentalLead
- /api/leads/create
- Calendar
- Gmail
- FloatingDentalAIChat
- Home
- siteServices.ts

## Validation checklist

- Patient list endpoint reads through ReadModelSourceProvider.
- Patient detail endpoint reads through ReadModelSourceProvider.
- Detail endpoint keeps Leads fallback for read-model misses.
- PatientManagementWorkspace keeps the existing list and mutation response shapes.
- No internal diagnostic fields are leaked in public patient API responses.
- Existing write endpoints remain outside the read-model pilot.
- Automated tests pass.
- Production build succeeds.

## Pilot result

15.5-A is ready for visual validation in PatientManagementWorkspace.
