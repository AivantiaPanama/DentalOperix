# 17.2-D.1 — Executive Dashboard API Test Reconciliation

## Objective

Restore the test suite after identifying that `executive-dashboard-api-readiness.test.ts` was asserting concrete route file presence from the readiness layer.

## Issue

The readiness test contained an assertion against the physical route tree for:

- `executive-observability/executive.ts`
- `executive-observability/operational.ts`
- `executive-observability/governance.ts`
- `executive-observability/snapshot.ts`

That assertion belongs to route implementation validation, not readiness validation. It also made the test sensitive to route file layout and path differences.

## Fix

The readiness test now validates that readiness reporting remains independent from concrete route files by checking:

- `routeImplementationIncluded` remains `false` in the readiness report.
- every readiness endpoint remains `planned`.
- the readiness source does not import route modules.
- the readiness source does not inspect `src/routes`.

## Architectural Result

- 17.2-C remains a readiness-only layer.
- 17.2-D route implementation remains separate.
- No public contracts changed.
- No UI was modified.
- No aggregate, adapter, read model, or write path was modified.

## Validation

- `npm test -- --run`: 261 passed
- `npm run build`: successful
