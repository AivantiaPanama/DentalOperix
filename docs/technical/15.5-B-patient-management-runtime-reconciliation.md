# 15.5-B — Patient Management Visual & Runtime Reconciliation

## Objective

Validate that `PatientManagementWorkspace` continues to operate correctly after Patient Management reads were migrated through the read-model source provider in 15.5-A.

## Scope

This phase is validation-only for runtime/UI compatibility. It does not change Patient Management UI behavior, write paths, lead creation, calendar, Gmail, booking, or site services.

## Architecture validation

`PatientManagementWorkspace` still consumes the existing stable API contract:

- `GET /api/patients/list` returns `{ success, patients }`.
- `patients` remains `PatientAdministrativeProfile[]`.
- Internal read-model diagnostics remain outside the public UI contract.
- Patient selection, search, status counts, and empty state all operate from the list response.

## Compatibility checks added

A jsdom runtime integration test now validates:

1. The workspace calls `/api/patients/list` with same-origin credentials.
2. The migrated list response renders patient cards and the selected administrative form.
3. Verified, pending, and incomplete status counters remain compatible.
4. Search continues to filter by administrative/read DTO fields.
5. Patient selection continues to update the detail form.
6. Empty list state remains compatible.
7. `resolvedIdentity` and `patientAggregateDiagnostics` are not rendered or leaked into the workspace.

## Files added

- `src/components/assistant/PatientManagementWorkspace.integration.test.tsx`

## Validation result

- `npm test -- --run`: passed, 185 tests.
- `npm run build`: passed.

## Conclusion

15.5-B confirms that Patient Management runtime behavior remains compatible with the read-model-backed list/detail migration completed in 15.5-A. No UI code changes were required.
