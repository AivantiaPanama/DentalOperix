# 17.3-F — Dashboard Integration Boundary Design

## Estado

IMPLEMENTED AS GOVERNANCE DESIGN ARTIFACT

Esta fase no implementa UI funcional, transporte, `fetch`, rutas, login ni API. Define únicamente límites de integración para una futura capa dashboard.

## Análisis arquitectónico

La arquitectura aprobada mantiene el flujo permitido:

```text
Future Dashboard UI
  -> Access Gate Boundary
  -> Data Client Boundary
  -> ExecutiveDashboardApiContracts
  -> Executive Dashboard API
```

La integración queda limitada a descriptores de frontera metric-only. No existe acceso directo a aggregates, adapters, read sources, read models ni telemetría cruda.

## Riesgos

### Riesgo: fallback accidental en cliente

Mitigación: la frontera declara `fallbackOwnership: server-read-platform-only` y mantiene `clientSideFallbackIncluded: false`.

### Riesgo: integración con transporte real antes de UI approval

Mitigación: la fase mantiene `runtimeStatus: design-contract-only`, `transportImplementationIncluded: false` y `fetchImplementationIncluded: false`.

### Riesgo: expansión hacia telemetría cruda

Mitigación: exposición exclusiva `metric-only` y prohibición explícita de eventos de telemetría.

### Riesgo: acoplamiento con login o almacenamiento de credenciales

Mitigación: se prohíbe `admin-login-credential-storage`, `browser-storage` y `credential-forwarding`.
