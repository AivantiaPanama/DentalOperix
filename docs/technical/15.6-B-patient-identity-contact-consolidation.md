# 15.6-B — Patient Identity & Contact Consolidation

## Objective

Consolidate Patient identity and contact read capabilities inside the Patient read aggregate while preserving the public API contract and the operational source-of-truth boundary.

## Architecture

The implemented read flow remains:

```text
GET /api/patients/list
GET /api/patients/:id
        ↓
ReadModelSourceProvider
        ↓
PatientAggregateReadService
        ↓
PatientIdentifierReadAdapter
PatientContactReadAdapter
PatientReadAdapter
        ↓
Read Models
        ↓
Fallback to Leads
```

## Implemented scope

- Added a dedicated `PatientIdentifierReadAdapter` for deterministic identity resolution.
- Added a dedicated `PatientContactReadAdapter` for primary email and phone resolution.
- Updated `PatientAggregateReadService` to use both adapters internally.
- Kept `ReadModelSource.patients` as `PatientAdministrativeProfile[]`.
- Kept `resolvedIdentity` and `resolvedContact` out of the public patient payload.
- Added diagnostics for contact consolidation.

## Identity rules

The identity priority remains:

```text
CID
Passport
FOREIGN_ID
TMP-PAT
```

If no explicit usable identifier exists, the service creates a deterministic `TMP-PAT-{patientId}` identity.

## Contact rules

The contact adapter resolves:

- primary email when available
- primary phone when available
- newest available contact when no primary flag exists
- empty values when no explicit contact exists

Empty contact resolution does not override fallback values already present in the administrative profile.

## Contract preservation

The public APIs remain unchanged:

```text
GET /api/patients/list -> { success, patients }
GET /api/patients/:id  -> { success, patient }
```

Internal diagnostics are not exposed through these endpoints.
