## Runtime Finding

Runtime testing showed that the admin dashboard could log:

```text
GET /api/goals/get 500 (Internal Server Error)
Falling back to localStorage for goal settings
```

The dashboard already had client-side fallback behavior, but the server-side API still emitted an avoidable HTTP 500 when sheet-backed goal settings were unavailable.

## Architectural Decision

The goals read endpoint must be resilient and non-blocking. When the sheet-backed goal settings source is unavailable, `/api/goals/get` now returns the certified default goal configuration with HTTP 200 instead of returning HTTP 500.

This preserves the existing client behavior while reducing noisy runtime errors.

## Implementation

Updated:

- `src/routes/api/goals/get.ts`
- `src/routes/api/goals.test.ts`

Behavior:

```text
Sheet goals available
  -> return sheet goals

Sheet goals unavailable
  -> return DEFAULT_GOAL_CONFIGURATION
  -> include fallback marker
  -> include warning message
  -> HTTP 200
```

Response shape on fallback:

```json
{
  "success": true,
  "goals": {
    "monthlyLeadsGoal": 50,
    "conversionGoal": 40,
    "attendanceGoal": 85,
    "pipelineValueGoal": 25000
  },
  "fallback": "default-goal-configuration",
  "warning": "Goal settings sheet unavailable; returned safe defaults."
}
```
