## Program Objective

Clinical Intelligence extends DentalOperix analytics from commercial and executive performance into patient and treatment execution analytics.

The program answers:

- How many patients have treatment plans?
- How many treatment plans are active or completed?
- How many treatment stages are completed?
- How much clinical outcome coverage exists?
- What data-quality gaps limit clinical interpretation?

## Architectural Position

```text
Clinical Read Models
  -> ClinicalReadAggregate
  -> ClinicalIntelligenceSnapshot
  -> Clinical Analytics API
  -> Executive / Operational Dashboards
```

Clinical Intelligence is read-only. It does not write clinical data, does not create a new master-data source, and does not alter the certified persistence architecture.

## Protected Components

The program must not modify:

- `BookingDialog`
- `processDentalLead`
- `/api/leads/create`
- Calendar integration
- Gmail integration
- `FloatingDentalAIChat`
- `Home`
- `siteServices.ts`

## Initial Scope

60.2 introduces:

- `ClinicalIntelligenceSnapshot`
- `createClinicalIntelligenceSnapshot()`
- `/api/analytics/clinical`
- `fetchClinicalIntelligence()`
- quality and governance limitations

## Out of Scope

- new clinical persistence tables
- clinical writes
- treatment plan mutation
- patient mutation
- source-of-truth changes
- revenue recognition
- billing ledger logic
- machine-learning diagnosis or treatment recommendation

## Governance Notes

Every 60.x output must identify itself as derived analytics. Clinical Intelligence v1 reports coverage and execution signals only; it does not make clinical decisions or modify patient records.

## 60.2-HF1 Revenue Dashboard Resilience Hotfix

STATUS: IMPLEMENTED
TYPE: Resilience Hotfix

Runtime testing identified that `/api/analytics/revenue?period=all` could return HTTP 500 and prevent the admin dashboard from loading Revenue Intelligence metrics.

The hotfix keeps Revenue Intelligence as the primary source and adds a controlled fallback inside `fetchRevenueDashboardMetrics()` to use `/api/crm/metrics` only when the Revenue Intelligence API is unavailable.

## 60.2-HF3 Goals API Resilience Hotfix

STATUS: IMPLEMENTED  
TYPE: Resilience Hotfix

Runtime testing identified that `/api/goals/get` could return HTTP 500 when sheet-backed goal settings were unavailable. The dashboard already fell back to localStorage, but the API error was noisy and avoidable.

The hotfix updates `/api/goals/get` so that unexpected sheet read failures return the safe default goal configuration with HTTP 200 and an explicit fallback marker.

Updated:

- `src/routes/api/goals/get.ts`
- `src/routes/api/goals.test.ts`
