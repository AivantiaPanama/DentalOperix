# 60.4-HF3 Clinic Email Delivery Guarantee

Status: IMPLEMENTED  
Type: Runtime booking notification hotfix  
Architecture change: NO  
Source of Truth change: NO  
Dual Write: NO

## Context

After 60.4-HF2 validation, the booking chain was partially confirmed:

```text
Lead Supabase .......... PASS
Clinic Calendar ........ PASS
Patient email .......... PASS
Clinic email ........... NOT VISIBLE IN CLINIC INBOX
```

The remaining incident was isolated to the clinic operational email notification. The lead source of truth and calendar event were not affected.

## Architectural Analysis

The certified lead architecture remains unchanged:

```text
Leads
  -> LeadPersistencePort
  -> LeadPersistenceProvider
  -> RelationalLeadPersistenceAdapter
  -> Supabase PostgreSQL
```

Google Calendar and Gmail remain downstream notification channels only. This hotfix does not make Gmail a source of truth and does not introduce Dual Write.

## Root Cause

When the clinic notification recipient resolves to the same account used as `GMAIL_SENDER`, Gmail can accept the outbound message but not surface it as a new incoming clinic inbox item. This means the API send can succeed while the clinic does not see a separate incoming email notification.

## Implementation

60.4-HF3 keeps the explicit clinic notification email introduced by 60.4-HF2 and adds clinic self-notification visibility handling:

```text
patient email send -> explicit patient notification
clinic email send  -> explicit clinic operational notification
if clinic recipient equals Gmail sender and Gmail returns a message id:
  attempt to mark the clinic message as INBOX + UNREAD
```

The Inbox/Unread mark is best-effort and only applies to the self-notification case. If Gmail scope does not allow message modification, the send result is preserved and the visibility mark failure is logged in development without rolling back lead persistence.

## Environment Notes

Recommended production/development configuration:

```text
GMAIL_SENDER=<authenticated clinic sender or approved alias>
CLINIC_NOTIFICATION_EMAIL=<clinic operational inbox>
```

If `CLINIC_NOTIFICATION_EMAIL` is the same mailbox as `GMAIL_SENDER`, clinic inbox visibility is improved when the OAuth token includes Gmail modify permission in addition to send permission:

```text
https://www.googleapis.com/auth/gmail.send
https://www.googleapis.com/auth/gmail.modify
```

If only `gmail.send` is granted, the clinic email is still sent through Gmail API, but Gmail may not display a self-addressed sender copy as a new Inbox item.
