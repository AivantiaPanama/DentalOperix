# 60.4-HF4 Unified Patient Confirmation Email

## Status

IMPLEMENTED / READY FOR USER VALIDATION

## Governance Context

60.4-HF4 was approved to revise Gmail and `processDentalLead` after user validation showed that patients were receiving two separate notifications:

1. A patient confirmation email.
2. A separate Calendar invitation / `invite.ics` email.

This hotfix preserves the certified Leads Source of Truth architecture:

```text
Leads
→ LeadPersistencePort
→ LeadPersistenceProvider
→ RelationalLeadPersistenceAdapter
→ Supabase PostgreSQL
```

It does not introduce Dual Write, Product Migration, Lead Replacement, a new source of truth, Analytics writes, or RBAC bypass.

## Objective

Ensure that the patient receives one consolidated email containing both:

- appointment confirmation content, and
- an attached `invite.ics` calendar file.

The clinic must continue receiving its operational notification separately, and the clinic Calendar event must continue to be created.

## Final Booking Notification Contract

When a public booking is confirmed:

1. The lead is persisted in Supabase through the certified persistence adapter.
2. The clinic Calendar event is created in the configured clinic calendar.
3. Calendar API duplicate attendee emails are suppressed by using `sendUpdates: "none"`.
4. The patient receives one Gmail confirmation email with:
   - appointment details,
   - service,
   - date and time,
   - link to the clinic Calendar event when available,
   - `invite.ics` attachment.
5. The clinic receives one operational Gmail notification.
6. Email delivery remains fail-soft after lead persistence and Calendar creation.

## Technical Notes

### Patient Email

The patient email now uses a multipart MIME message:

```text
multipart/mixed
├─ text/html confirmation body
└─ text/calendar invite.ics attachment
```

The `.ics` file uses appointment data from the same booking payload and is attached directly to the confirmation email.

### Calendar Event

The clinic Calendar event remains created with the patient and clinic as attendees for event context, but Calendar API email delivery is disabled with `sendUpdates: "none"` to avoid duplicate patient notifications.

### Clinic Email

The clinic notification remains a separate operational message so the clinic has a clear internal alert independent of the patient-facing confirmation.
