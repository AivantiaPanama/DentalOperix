- Executive Dashboard architecture.
- Product model.
- Calendar or Gmail integrations.

## Runtime Activation Model

Relational runtime activation requires all of the following flags:

```env
LEADS_PERSISTENCE_MODE=relational-db
RELATIONAL_CUTOVER_APPROVED=true
RELATIONAL_RUNTIME_ACTIVATION_APPROVED=true
DATABASE_URL=postgresql://...
```

If any required flag is missing, `RelationalLeadPersistenceAdapter` fails closed with `LeadPersistenceNotConfiguredError`.

## Adapter Capabilities

When all runtime flags are enabled, the relational adapter supports:

- `appendLead()` → INSERT into `leads`.
- `updateLead()` → UPDATE `status`, `calendar_event_id`, and `email_sent`.
- `listLeads()` → SELECT from `leads` ordered by `created_at` descending.
- `getHealth()` → reports active/writable only when all 57.8 flags are present.

## Cutover Procedure

1. Confirm Supabase connectivity:

```bash
npm run validate:relational-connectivity -- .env.relational.local
```

2. Confirm schema:

```bash
npm run validate:relational-schema -- .env.relational.local
```

3. Confirm dry-run:

```bash
npm run validate:relational-dry-run -- .env.relational.local
```

4. Prepare controlled runtime flags:

```env
LEADS_PERSISTENCE_MODE=relational-db
RELATIONAL_CUTOVER_APPROVED=true
RELATIONAL_RUNTIME_ACTIVATION_APPROVED=true
```

5. Validate flags:

```bash
npm run validate:relational-runtime-flags -- .env.relational.local
```

6. Deploy during the approved cutover window.
