## Purpose

57.8-C defines the controlled production activation procedure for moving the active physical persistence for Leads from Google Sheet to Supabase PostgreSQL.

This is a physical persistence transition only. It does not change:

- Leads as Source of Truth.
- Lead domain ownership.
- Product model.
- Read Model Platform v2.
- Executive Dashboard architecture.
- Calendar or Gmail integrations.

## Entry Criteria

All entry criteria must be complete before changing runtime flags:

```text
Google Sheet backup/export: COMPLETED
DEV Supabase connectivity: PASS
DEV Supabase schema: PASS
DEV dry-run validation: PASS
PROD Supabase connectivity: PASS
PROD Supabase schema: PASS
PROD dry-run validation: PASS
Rollback runbook: AVAILABLE
Executive approval: GRANTED
Controlled window: SCHEDULED
```

## Runtime Activation Flags

Production runtime activation requires all flags below:

```env
LEADS_PERSISTENCE_MODE=relational-db
RELATIONAL_CUTOVER_APPROVED=true
RELATIONAL_RUNTIME_ACTIVATION_APPROVED=true
RELATIONAL_PRODUCTION_CUTOVER_READINESS_APPROVED=true
```

The relational adapter remains fail-closed if any required runtime activation flag is absent.

## Pre-Cutover Validation

Run against the production relational environment file:

```bash
npm run validate:production-cutover-readiness -- .env.relational.prod
```

Expected result:

```text
Result: PASS
```

This validation checks:

- Runtime flags are explicitly set for the controlled window.
- Production database URL is configured.
- Google Sheet rollback reference is configured.
- Required tables exist.
- Required Leads columns exist.
- Current Leads and audit row counts can be read.

This validation performs no writes.

## Controlled Activation

During the approved production window only:

1. Confirm Google Sheet backup/export timestamp.
2. Confirm `.env.relational.prod` contains the production Supabase `DATABASE_URL`.
3. Set the runtime activation flags.
4. Deploy the runtime environment.
5. Start the application.
6. Confirm `LeadPersistenceProvider` selects `relational-db`.

## Post-Cutover Validation

After runtime activation, run:

```bash
npm run validate:production-post-cutover -- .env.relational.prod
```

This script requires:

```env
RELATIONAL_POST_CUTOVER_VALIDATION_APPROVED=true
```

The validation performs a synthetic transactional check:

- INSERT synthetic Lead.
- SELECT synthetic Lead.
- UPDATE synthetic Lead.
- INSERT audit row.
- ROLLBACK cleanup.
- Confirm persisted synthetic rows after rollback = 0.

Expected result:

```text
Result: PASS
```

## Rollback Trigger Matrix

Rollback must be declared immediately if any of the following occur:

```text
Write failure
Read failure
Data integrity failure
Database connectivity failure
Audit failure
Unexpected dual-write behavior
Governance violation
Operational instability during watch window
```

## Rollback Procedure

Set:

```env
LEADS_PERSISTENCE_MODE=google-sheet
RELATIONAL_CUTOVER_APPROVED=false
RELATIONAL_RUNTIME_ACTIVATION_APPROVED=false
RELATIONAL_PRODUCTION_CUTOVER_READINESS_APPROVED=false
RELATIONAL_POST_CUTOVER_VALIDATION_APPROVED=false
```

Then restart/redeploy the runtime and validate Google Sheet operations.

## Governance Invariants

The cutover must preserve:

```text
Leads = Source of Truth
No Dual Write
No Multiple Sources of Truth
No Projection Engine
No Product Migration
No Lead Replacement
```
