## Status

## Type

## Problem

Manual runtime validation showed the dashboard could display the empty CRM message on first navigation even when backend endpoints were already returning real data:

```text
/api/analytics/revenue?period=all → source: google-sheets, totals.leads = 18
/api/crm/metrics → real metrics payload
```

The issue was isolated to dashboard first-load UI state, not Google Sheets connectivity, RBAC, session validation, or Revenue Intelligence data generation.

## Scope

Updated:

```text
src/routes/admin/dashboard.tsx
src/routes/admin/-dashboard.test.tsx
```

No changes were made to protected components or ingestion paths.

## Implementation

The dashboard now:

1. Tracks the latest metrics request with a request id.
2. Treats an initial `emptyCRM` snapshot as a candidate for reconciliation.
3. Performs one immediate reconciliation fetch before exposing the empty CRM UI state.
4. Suppresses the empty-state banner while reconciliation is active.
5. Keeps the latest successful request only, avoiding stale first-load state overwrites.

## Expected runtime behavior

On first navigation to `/admin/dashboard`:

- If Revenue/CRM metrics return real leads, metrics render normally.
- If an initial empty snapshot is transient, the dashboard reconciles before showing empty state.
- If the CRM truly has zero leads after reconciliation, the empty CRM message remains valid.

## Validation recommendation

Manual validation:

```text
1. Open /admin/dashboard in a fresh browser session.
2. Confirm the dashboard does not display “Aún no existen registros en el CRM” when endpoints return leads.
3. In console, verify:
   fetch('/api/analytics/revenue?period=all').then(r => r.json())
   returns source: google-sheets and totals.leads > 0.
4. Confirm refresh is no longer required to show metrics.
```
