# 57.7-D Relational Dry-Run Validation

## Status

COMPLETED AS IMPLEMENTATION PACKAGE / READY FOR LOCAL EXECUTION

## Governance Classification

Documentation and validation tooling only.

This phase does not authorize cutover and does not activate relational persistence.

## Purpose

Validate that the deployed Supabase relational schema can support controlled write/read/update/audit operations using synthetic data while preserving the existing production persistence posture.

## Required Runtime State

```text
LEADS_PERSISTENCE_MODE=google-sheet
RELATIONAL_CUTOVER_APPROVED=false
RELATIONAL_DRY_RUN_APPROVED=true
RELATIONAL_DRY_RUN_ENABLED=true
```

## Validated Operation Model

The dry-run script validates the following operations inside an explicit database transaction:

1. Insert one synthetic lead.
2. Read the synthetic lead.
3. Update the synthetic lead.
4. Insert one synthetic migration audit record.
5. Roll back the transaction.
6. Confirm that the synthetic lead is not persisted after rollback.

## Data Policy

The dry-run uses synthetic data only.

No production Google Sheet data is read, copied, migrated, or written to Supabase by this validation.

## Governance Controls

The dry-run script refuses execution if:

- `LEADS_PERSISTENCE_MODE` is not `google-sheet`.
- `RELATIONAL_CUTOVER_APPROVED` is `true`.
- `RELATIONAL_DRY_RUN_APPROVED` is not `true`.
- `RELATIONAL_DRY_RUN_ENABLED` is not `true`.
- `DATABASE_URL` or `POSTGRES_URL` is missing.

## Commands

## Expected Output

```text
Operations validated: INSERT lead, SELECT lead, UPDATE lead, INSERT audit, ROLLBACK cleanup
Persisted rows after rollback: 0
Result: PASS
```

## Current Production Posture

```text
Google Sheet = Active Physical Persistence
Relational Database = Prepared Future Physical Persistence

```

## Result Criteria

PASS means relational infrastructure is capable of controlled synthetic transactional operations.

PASS does not mean cutover is authorized.
