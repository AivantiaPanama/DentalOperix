## Governing Document

The certified 57.x persistence architecture remains authoritative:

```text
Leads
  -> LeadPersistencePort
  -> LeadPersistenceProvider
  -> RelationalLeadPersistenceAdapter
  -> Supabase PostgreSQL
```

58.3 does not modify the certified persistence chain. The integration layer is read-only analytics and does not create a new source of truth.

## Purpose

58.3 exposes the 58.2 `RevenueSnapshotV1` contract through a stable read-only integration boundary so dashboard and reporting surfaces can consume one official Revenue Intelligence payload instead of duplicating KPI formulas.

## Implemented Files

- `src/routes/api/analytics/revenue.ts`
- `src/routes/api/analytics/revenue.test.ts`
- `src/lib/api/revenue-intelligence.ts`

## Read-Only Endpoint

```text
GET /api/analytics/revenue
GET /api/analytics/revenue?period=<dashboard-period>
```

The endpoint returns:

```ts
{
  success: true,
  period: string | null,
  snapshot: RevenueSnapshotV1
}
```

The endpoint uses the existing dashboard period normalization and filtering rules through:

- `normalizeDashboardPeriod`
- `filterLeadsByPeriod`

## Authorization

The endpoint requires:

```text
reports:read
```

Unauthorized and forbidden responses follow the existing RBAC response helpers.

## Production Guard

The endpoint follows the existing CRM metrics production access guard. If the production Google token is unavailable, the endpoint returns a restricted-access response instead of attempting to fetch source data.

## Client Adapter

58.3 adds a client adapter:

```ts
fetchRevenueIntelligence(period?: string | null)
```

Location:

```text
src/lib/api/revenue-intelligence.ts
```

This adapter is intentionally separate from dashboard UI alignment. Dashboard consumption is reserved for a future 58.4 phase to avoid coupling the endpoint introduction with UI changes.
