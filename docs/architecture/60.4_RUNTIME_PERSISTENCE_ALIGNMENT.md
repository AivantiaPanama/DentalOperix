## Problem Statement

The 60.4 audit identified drift between the certified persistence documentation and runtime code paths.

Observed drift:

- `processDentalLead` wrote directly to Google Sheets through CRM helpers.
- `/api/leads/list` read directly from Google Sheets.
- derived analytics and operational repositories read directly from Google Sheets.
- `.env.relational.prod` existed as a real environment file in the distributed package.

This created ambiguity about whether Google Sheets remained active physical persistence after the certified 57.x cutover.

## Remediation Scope

60.4 aligns runtime access to the certified persistence path without introducing dual write or a new source of truth.

Implemented containment and alignment:

1. Removed the real `.env.relational.prod` file from the working tree.
2. Changed default active persistence selection to `relational-db`.
3. Routed lead creation and post-creation updates through `LeadPersistenceProvider`.
4. Routed `/api/leads/list` through `LeadPersistenceProvider`.
5. Routed read-only analytics and operational read models through `LeadPersistenceProvider`.
6. Restricted Google Sheets persistence to explicit rollback mode only.

## Runtime Selection Rules

### Certified default

When no override is provided, active runtime persistence resolves to:

```text
LeadPersistenceProvider -> RelationalLeadPersistenceAdapter
```

### Relational activation

Relational runtime remains fail-closed unless the certified 57.8 runtime flags are enabled:

```text
LEADS_PERSISTENCE_MODE=relational-db
RELATIONAL_CUTOVER_APPROVED=true
RELATIONAL_RUNTIME_ACTIVATION_APPROVED=true
```

### Google Sheets rollback

Google Sheets is no longer the default active persistence. It can be selected only as an explicitly approved rollback path:

```text
LEADS_PERSISTENCE_MODE=google-sheet
GOOGLE_SHEETS_ROLLBACK_APPROVED=true
```

Rollback mode must not be used for dual write, analytics writes, or parallel source-of-truth behavior.
