# 57.8 Rollback Runbook

## Status

```text
STATUS: ACTIVE ROLLBACK RUNBOOK FOR CONTROLLED CUTOVER
```

## Rollback Principle

Rollback returns Leads physical persistence to Google Sheet without changing Leads as Source of Truth.

## Rollback Triggers

Rollback must be declared if any of the following occur:

- Relational writes fail.
- Relational reads fail.
- Data integrity validation fails.
- Calendar/Gmail post-lead update validation fails.
- Unexpected dual-write behavior appears.
- Operational owner declares NO-GO during cutover window.

## Rollback Procedure

1. Stop the application process or deployment slot using relational flags.
2. Restore flags:

```env
LEADS_PERSISTENCE_MODE=google-sheet
RELATIONAL_CUTOVER_APPROVED=false
RELATIONAL_RUNTIME_ACTIVATION_APPROVED=false
```

3. Restart the application.
4. Validate Google Sheet write path with the existing operational smoke test.
5. Validate `Google Sheet = Active Persistence` in health output.
6. Document the incident and preserve Supabase state for forensic review.

## Rollback Validation

Required post-rollback checks:

```text
appendLead path = Google Sheet
updateLead path = Google Sheet
listLeads path = Google Sheet
Calendar integration unaffected
Gmail integration unaffected
```

## Governance Controls

Rollback does not authorize:

- Dual Write.
- Multiple Sources of Truth.
- Projection Engine.
- Product Migration.
- Read Model Platform reopening.
