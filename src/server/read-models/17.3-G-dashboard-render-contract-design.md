# 17.3-G — Dashboard Render Contract Design

## Status

APPROVED AND IMPLEMENTED AS DESIGN CONTRACTS ONLY.

This phase defines render contracts for the future Executive Dashboard UI without implementing visual components, routes, fetch clients, transport, login, API behavior, persistence, fallback, or domain logic.

## Architectural Scope

The render contract layer is a presentation boundary only. It may describe how future dashboard surfaces represent request states, but it must not compute platform health, domain health, aggregate health, governance status, fallback, or authorization.

Allowed flow:

```text
Future Dashboard UI
  -> Render Contract
  -> Integration Boundary
  -> Data Client Design
  -> ExecutiveDashboardApiContracts
```

Forbidden flows:

```text
Render Contract -> Aggregates
Render Contract -> Adapters
Render Contract -> Raw Telemetry
Render Contract -> Read Sources
Render Contract -> Read Models directly
Render Contract -> Fallback Logic
Render Contract -> Persistence
```

## Render States

The only approved render states are:

- loading
- ready
- empty
- error
- forbidden

Only the `ready` state may display authorized metric values. All other states are presentation-only and must not expose raw telemetry, aggregate state, adapter state, credentials, internal exceptions, source data, or fallback internals.

## Dashboard Surfaces

The render contract applies to exactly three surfaces:

- Executive Dashboard
- Operational Dashboard
- Governance Dashboard

Each surface remains bound to:

- `ExecutiveDashboardApiContracts`
- `executive-observability:read`
- `metric-only` exposure
- `GET` endpoint descriptors inherited from the integration boundary
